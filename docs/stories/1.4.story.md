# Story 1.4 — ADR Registry

## User Story
Como agente AIOS, quero registrar decisões arquiteturais formalmente, para que todo time de IA atual e futuro saiba o que foi decidido, por quê, e o que foi descartado.

## Complexidade
**S (Pequeno)** — ADRs são principalmente arquivos Markdown + CRUD simples no Postgres.

## Riscos
- Numeração sequencial requer coordenação entre agentes — o serviço deve ser a fonte de verdade para o próximo número disponível (não deixar à cargo do agente calcular)
- AC5 ("processo integrado ao fluxo") é regra de processo, não entrega técnica — reformulado como SOP obrigatório

## Technical Context

### Data Model — ADR (de `docs/architecture.md` §4)

```typescript
interface ADR {
  id: string;
  number: number;         // sequencial: 001, 002...
  title: string;
  status: 'proposed' | 'accepted' | 'deprecated' | 'superseded';
  context: string;        // Por que a decisão foi necessária
  decision: string;       // O que foi decidido
  rationale: string;      // Por que esta escolha
  alternatives: string[]; // O que foi descartado e por quê
  consequences: string;   // Impactos da decisão
  createdBy: string;      // agentId
  storyId: string | null;
  filePath: string;       // Path do arquivo .md
  createdAt: string;
  updatedAt: string;
}
```

### Template de ADR (conteúdo de `ADR-TEMPLATE.md`)

```markdown
# ADR-NNN — Título da Decisão

**Status:** Proposed | Accepted | Deprecated | Superseded
**Data:** YYYY-MM-DD
**Autor:** @agent-id
**Story:** X.Y (se aplicável)

## Contexto
[Por que esta decisão foi necessária? Qual problema resolve?]

## Decisão
[O que foi decidido, de forma objetiva.]

## Rationale
[Por que esta escolha? Quais os critérios usados?]

## Alternativas Descartadas
| Alternativa | Razão de Descarte |
|-------------|------------------|
| Opção A     | ...              |
| Opção B     | ...              |

## Consequências
[Impactos positivos e negativos desta decisão.]

## Referências
- Link para story, PRD, ou outro artefato relacionado
```

### API Endpoints (Hono — `packages/core`)
```
GET  /api/adrs                # Lista todas as ADRs (ordenadas por número)
GET  /api/adrs/:number        # ADR específica por número
POST /api/adrs                # Registra nova ADR (cria arquivo .md + persiste no Postgres)
GET  /api/adrs/next-number    # Retorna próximo número disponível (fonte de verdade para numeração)
```

### Schema Postgres
```sql
CREATE TABLE adrs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  number INTEGER NOT NULL UNIQUE,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'proposed',
  context TEXT,
  decision TEXT,
  rationale TEXT,
  alternatives JSONB DEFAULT '[]',
  consequences TEXT,
  created_by TEXT,
  story_id TEXT,
  file_path TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Notas para @dev
- ADR-001 deve ser a primeira criada — documentar a decisão do monorepo pnpm
- Numeração: sempre 3 dígitos com zero-padding (001, 002, ..., 010, 011...)
- Arquivo .md é a fonte de verdade — Postgres é índice para busca rápida por agentes
- Quando `POST /api/adrs` é chamado: (1) cria o arquivo .md em `docs/adrs/`; (2) persiste metadados no Postgres
- ADRs são write-once (exceto `status` e `updatedAt`) — não modificar decisões passadas
