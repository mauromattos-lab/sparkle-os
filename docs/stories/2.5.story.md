# Story 2.5 — Protocolo de Melhoria Incremental dos Fluxos

## User Story
Como agente AIOS, quero um protocolo formal para propor e aplicar melhorias nos fluxos n8n da Zenya de forma incremental — node a node, fluxo a fluxo — para que mudanças sejam seguras, rastreáveis e revertíveis sem risco à operação atual de clientes.

## Riscos
- O protocolo pode ser demasiado burocrático para melhorias menores — calibrar os gates para não travar evolução do sistema
- A distinção entre mudança de IP (requer Mauro) e mudança funcional (pode ser delegada) precisa ser clara — ambiguidade aqui gera conflito de processo

### Referência: IP vs. Funcional
[Source: docs/zenya/ip/ZENYA-PROMPTS.md, docs/zenya/ip/ZENYA-LOGIC.md]
- **Toca IP:** mudanças em `systemMessage` (prompts), código JS em nós `Mensagem encavalada?` ou `Velocidade digitação` → gate Mauro obrigatório
- **Não toca IP:** mudanças em parâmetros de schedule, ajuste de URLs, adição de node de log, configuração de credenciais → gate de @architect suficiente

### Modelo de Ambiente de Teste
[Source: docs/zenya/NUCLEUS-CONTRACT.md + Story 2.4 — POST /flows/:id/clone]
```
1. Clone do fluxo: POST /nucleus/zenya/flows/{id}/clone
2. Configurar clone para apontar para inbox de teste no Chatwoot
3. Testar com número de WhatsApp de teste
4. Se validado: aplicar mudança no fluxo de produção
5. Deletar clone de teste
```

### Estrutura do SOP
O SOP deve cobrir os seguintes estágios:
```
PROPOSTA → CLASSIFICAÇÃO → APROVAÇÃO → IMPLEMENTAÇÃO (clone) → VALIDAÇÃO → PROMOÇÃO → CHANGELOG
```
Com bifurcação no gate de aprovação conforme a classe da mudança.
