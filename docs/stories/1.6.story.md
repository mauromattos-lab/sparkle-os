# Story 1.6 — Protocolo de Escalação para Mauro

## User Story
Como sistema SparkleOS, quero critérios claros de quando agir autonomamente vs. escalar para Mauro, para que ele seja acionado apenas quando genuinamente necessário.

## Technical Context

### Classificação de Decisões

| Tipo | Quem decide | Exemplos |
|------|-------------|---------|
| **Autônoma** | Agente | Escolha de biblioteca, estrutura de código, estratégia de teste |
| **Documentada** | Agente + ADR | Decisão arquitetural, mudança de stack, novo padrão |
| **Recomendação** | Agente propõe, Mauro aprova | Mudança de infraestrutura, novo serviço pago, migração |
| **Escalação obrigatória** | Mauro decide | IP da Zenya, aprovação de épico, contratação de serviço caro |

### Triggers de Escalação Obrigatória
1. Qualquer mudança no IP da Zenya (lore, personalidade, assets visuais)
2. Novo custo mensal > $50 adicionado à infraestrutura
3. Acesso a dados de clientes ativos da Zenya
4. Mudança de arquitetura que impacta múltiplos Épicos
5. Decisão que não tem consenso entre agentes (conflito de recomendações)
6. Qualquer operação destrutiva em produção (drop de tabela, delete em massa)

### Data Model — DecisionQueue
```typescript
interface PendingDecision {
  id: string;
  title: string;
  context: string;        // Por que Mauro precisa decidir
  options: Array<{
    label: string;
    description: string;
    recommendation: boolean; // o agente recomenda esta?
    pros: string[];
    cons: string[];
  }>;
  requestedBy: string;    // agentId
  storyId: string | null;
  priority: 'urgent' | 'normal' | 'low';
  status: 'pending' | 'resolved' | 'deferred';
  resolution: string | null;
  createdAt: string;
  resolvedAt: string | null;
}
```

### API Endpoints (Hono — `packages/core`)
```
GET  /api/decisions/pending     # Lista decisões aguardando Mauro
POST /api/decisions             # Agente registra nova decisão pendente
PATCH /api/decisions/:id        # Mauro resolve (via Piloting Interface no Epic 4)
```

### Schema Postgres
```sql
CREATE TABLE pending_decisions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  context TEXT NOT NULL,
  options JSONB NOT NULL DEFAULT '[]',
  requested_by TEXT NOT NULL,
  story_id TEXT,
  priority TEXT NOT NULL DEFAULT 'normal',
  status TEXT NOT NULL DEFAULT 'pending',
  resolution TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  resolved_at TIMESTAMPTZ
);
```
