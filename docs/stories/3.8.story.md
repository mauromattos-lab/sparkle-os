# Story 3.8 — DNA de Mauro

## User Story

Como Cérebro Coletivo do SparkleOS, quero ter acesso ao DNA de Mauro — seus princípios, preferências e filtros de raciocínio — registrado diretamente por ele em sessão de co-criação, para que eu possa aplicar e priorizar conhecimento com o estilo decisório de Mauro como âncora, garantindo que insights com `source='mauro_input'` tenham `confidenceLevel='authoritative'` e nunca sejam rejeitados automaticamente.

## Acceptance Criteria

1. **Sessão de co-criação conduzida** com Mauro diretamente — @analyst facilita sessão estruturada com roteiro definido nos Analyst Notes. Nenhum princípio é extraído de documentos antigos. Mauro é a fonte primária e única.

2. **Documento `docs/dna/mauro-dna.md` criado** com estrutura completa: (a) Princípios Fundacionais (mínimo 5, máximo 15), (b) Filtros de Raciocínio (como Mauro avalia trade-offs e toma decisões), (c) Preferências de Trabalho (estilo de colaboração, comunicação, ritmo), (d) Anti-padrões (o que Mauro explicitamente rejeita), (e) Exemplos de decisões passadas que ilustram os princípios.

3. **Mínimo de 15 insights ingeridos** no Cérebro via `POST /brain/insights` com `source='mauro_input'`. Cada insight deve ter: `content` com o princípio/filtro em linguagem natural, `source='mauro_input'`, `confidenceLevel='authoritative'` (automático pela API conforme arquitetura), `tags` relevantes (ex: `["dna", "principio", "filtro-raciocinio"]`).

4. **Verificação de recuperação:** `BrainClient.search()` retorna pelo menos 3 insights de DNA de Mauro para cada uma das seguintes queries de teste: (a) `"como priorizar quando há múltiplas demandas"`, (b) `"qualidade vs velocidade de entrega"`, (c) `"como tomar decisões sem informação completa"`. Resultados devem ter `similarity >= 0.70`.

5. **Filtros de raciocínio ativos:** Pelo menos 5 insights com tag `"filtro-raciocinio"` ingeridos e recuperáveis via `BrainClient.getContext()` — estes são os filtros que guiam como agentes AIOX priorizam e aplicam conhecimento ao contexto de Mauro.

6. **SOP criado** em `docs/sops/sop-dna-mauro.md` — como consultar o DNA de Mauro via `BrainClient`, quando usar `source='mauro_input'` ao ingerir novos insights de Mauro, e como os agentes devem interpretar resultados com `confidenceLevel='authoritative'`.

7. **Script de carga** em `docs/dna/load-mauro-dna.ts` (ou `.sh`) documentando os calls exatos de ingestão — reproduzível se o ambiente precisar ser recarregado. Não precisa ser executável via CI, apenas servir como registro auditável dos insights carregados.

## Complexidade

**S (Pequeno)** — Nenhuma modificação de código de produção. O esforço está na qualidade da facilitação e na profundidade da co-criação. A ingestão usa a interface já construída na Story 3.5 (`BrainClient.ingest()`). O risco é de conteúdo (DNA mal capturado), não de implementação.

## Riscos

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| DNA superficial ou genérico | Filtros de raciocínio imprecisos — agentes tomam decisões desalinhadas com Mauro | Roteiro de co-criação aprofundado (ver Analyst Notes) — perguntas que exploram casos concretos, não abstratos |
| Mauro expressa princípios contraditórios | Insights conflitantes no Cérebro | @analyst mapeia tensões durante sessão — Mauro decide qual princípio prevalece em qual contexto |
| Princípios muito abstratos (difíceis de recuperar por vector search) | `similarity` baixo nas queries de teste — AC4 falha | Cada princípio deve ter: enunciado direto + exemplo concreto + anti-exemplo. Isso aumenta a densidade semântica. |
| BrainClient.ingest() indisponível (Story 3.5 não Done) | Bloqueio de ingestão | Dependência explícita em 3.5 Done. Fallback: chamadas HTTP diretas ao `POST /brain/insights`. |
| Sessão interrompida — DNA incompleto | AC1 não atendido | Script de carga auditável (`load-mauro-dna.ts`) permite retomar sem perda. Sessão pode ser dividida em múltiplas partes. |

## Analyst Notes

### Metodologia de Co-criação

Esta story é uma **sessão de elicitação de conhecimento**, não de desenvolvimento de software. @analyst atua como facilitador e escriba — Mauro é o especialista e fonte.

**Princípio fundamental (FR15):** Nenhum princípio pode ser inferido de documentos anteriores. Toda informação vem da boca de Mauro na sessão.

### Roteiro da Sessão (estimativa: 60–90 min)

**Bloco 1 — Contexto e Aquecimento (10 min)**
- Explicar o propósito: "O DNA de Mauro vai guiar como o Cérebro prioriza conhecimento. Tudo que você disser aqui tem peso máximo — authoritative."
- Perguntar: "Quais são as 3 decisões mais importantes que você tomou no SparkleOS até hoje? O que guiou cada uma?"

**Bloco 2 — Princípios Fundacionais (20 min)**
- "Se você tivesse que ensinar alguém a pensar como você em 5 frases, quais seriam?"
- "O que nunca pode ser comprometido, não importa a pressão?"
- "Qual é o princípio que mais te diferencia de outros founders/líderes que você conhece?"
- Para cada princípio: pedir um exemplo concreto + uma situação onde o princípio foi testado.

**Bloco 3 — Filtros de Raciocínio (20 min)**
- "Quando você precisa escolher entre velocidade e qualidade, como você decide?"
- "Quando você tem informação incompleta e precisa agir, qual é seu processo?"
- "Como você prioriza quando tudo parece urgente ao mesmo tempo?"
- "Quando você sente que está tomando a decisão errada, o que te faz parar?"
- Para cada resposta: "Isso sempre se aplica ou existem exceções?"

**Bloco 4 — Preferências de Trabalho (15 min)**
- "Como você prefere receber informações dos agentes — detalhado ou resumido?"
- "Quando um agente deve agir autonomamente vs. te perguntar?"
- "O que te irrita quando um sistema ou pessoa está te ajudando?"
- "Como você sabe quando um trabalho está bom o suficiente para avançar?"

**Bloco 5 — Anti-padrões e Validação (15 min)**
- "O que você definitivamente NÃO quer que o sistema faça, mesmo que pareça eficiente?"
- "Quais erros você já cometeu que o sistema deve ajudar a não repetir?"
- Revisar os princípios capturados: "Isso representa você? Falta algo importante?"

### Formato de Cada Insight para Ingestão

```typescript
// Exemplo de insight de DNA de Mauro

Quando há pressão por prazo, cortar escopo (não adicionar dívida técnica). Exemplo: Em Epic 2, optamos por fazer menos features com qualidade total em vez de mais features frágeis.",

});
```

### Verificação de Recuperação (AC4)

Após ingestão, executar as 3 queries de teste com `BrainClient.search()`:

```typescript
const queries = [
  "como priorizar quando há múltiplas demandas",
  "qualidade vs velocidade de entrega",
  "como tomar decisões sem informação completa"
];

for (const query of queries) {
  const { results } = await brainClient.search(query, {
    statusFilter: ['validated', 'applied'],
    threshold: 0.70,
    limit: 5
  });
  const dnResults = results.filter(r => r.source === 'mauro_input');
  // AC4: dnResults.length >= 3 para cada query
}
```

**Nota:** Insights com `source='mauro_input'` têm `confidenceLevel='authoritative'` automaticamente (arquitetura do Cérebro). Isso significa que nunca são rejeitados pelo validador de qualidade e têm prioridade máxima no ranking de resultados.
