# Story 3.3 — Validação de Qualidade do Conhecimento

## User Story
Como Cérebro Coletivo, quero um mecanismo de validação antes de qualquer aplicação de insight, para que o Cérebro não acumule ruído que contamina decisões futuras.

## Acceptance Criteria

- [x] 1. `PATCH /brain/insights/{id}/validate` implementado: calcula ou recebe `qualityScore` (0.0–1.0), aplica critérios de qualidade, transiciona `raw → validated` quando `qualityScore >= 0.6` (threshold baseline), preenche `validatedAt`, `validatedBy`, `validationNotes`
- [x] 2. Critérios de qualidade documentados e implementados como função em `packages/brain/src/services/quality.service.ts` — mínimo 3 dimensões: (a) especificidade (conteúdo é específico e acionável vs. genérico), (b) evidência (insight baseado em dado real vs. suposição), (c) relevância (relação clara com operação SparkleOS)
- [x] 3. Insights com `qualityScore < 0.6` NÃO são rejeitados automaticamente — permanecem com `status='raw'` e `validationNotes` explicando o gap. Só chegam a `rejected` via chamada explícita a `PATCH /brain/insights/{id}/reject`
- [x] 4. `PATCH /brain/insights/{id}/reject` implementado: `reason` obrigatório, `status → 'rejected'`, insight não é deletado
- [x] 5. `POST /brain/insights/search` implementado: busca semântica com threshold 0.75, retorna apenas `validated` e `applied`, resultado inclui `source` e `confidenceLevel`. Aceita filtros opcionais: `minConfidence`, `statusFilter`, `threshold`, `limit`
- [x] 6. SOP criado em `docs/sops/sop-revisar-promover-insights.md` — processo para revisar insights em `raw`, promovê-los para `validated` ou rejeitá-los, incluindo critérios de decisão por `confidenceLevel` de fonte

## Complexidade

## Riscos

- **Threshold 0.6 pode ser inadequado:** Valor baseline definido aqui — deve ser revisável. Deve ser configurável (ex: constante em `quality.service.ts`) para ajuste após dados reais
- **Busca semântica com base pequena:** Com poucos insights validados, resultados podem ser de baixa qualidade — threshold 0.75 pode não retornar nada. Endpoint deve retornar array vazio sem erro
- **`depends_on: 3.2`:** Esta story requer a tabela `insights` e o endpoint `POST /brain/insights` funcionais (Story 3.2). Não implementar antes do schema estar criado

### Quality Service — 3 Dimensões

```typescript
// packages/brain/src/services/quality.service.ts

interface QualityCriteria {
  specificity: number;   // 0.0–1.0: específico vs. genérico
  evidence: number;      // 0.0–1.0: baseado em dado real vs. suposição
  relevance: number;     // 0.0–1.0: relevante para operação SparkleOS
}

// Score final = média das dimensões (pesos iguais para v1)
function calculateQualityScore(criteria: QualityCriteria): number {
  return (criteria.specificity + criteria.evidence + criteria.relevance) / 3;
```
