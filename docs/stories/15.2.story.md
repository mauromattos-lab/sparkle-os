## Descrição

Criar `packages/zenya/scripts/smoke-template.mjs` — script-modelo que qualquer tenant novo pode copiar, renomear, e adaptar cenários. Objetivo: eliminar "cada tenant reinventa smoke do zero" e alinhar o método ao passo 3 do playbook (Story 15.1).

**Fonte:** `packages/zenya/scripts/smoke-scar.mjs` (214 linhas, criado durante onboarding Scar AI). Já tem a estrutura certa. Precisa virar template genérico com comentários explicativos + 2 variantes (prompt-only e prompt+KB).

## Acceptance Criteria

1. **Arquivo `packages/zenya/scripts/smoke-template.mjs` criado**, derivado de `smoke-scar.mjs` com as seguintes diferenças:
   - Nome do arquivo genérico (não referencia Scar/PLAKA)
   - Variável `CHATWOOT_ACCOUNT_ID` sem default — obrigatório passar via `--tenant=<id>` ou env (se omitido, erro com mensagem clara)
   - Comentário de header explicando: propósito, quando usar, como adaptar
   - Cenários de exemplo NÃO hardcoded — são marcados como `// TODO: adapte pro seu tenant — exemplo abaixo é de PLAKA`, com comentário sugerindo derivar da fonte (prompt + KB + portfólios)
   - 2 seções de cenários separadas por comentário de bloco:
     - `// === VARIANTE A: TENANT PROMPT-ONLY (ex: Scar AI) ===` com 3-4 cenários genéricos (idioma, objeção, fechamento crítico)
     - `// === VARIANTE B: TENANT COM KB (ex: PLAKA) ===` com 3-4 cenários genéricos (hit, miss, edge case de fuzzy)
   - Usuário final ativa uma das variantes descomentando bloco + ajustando conteúdo específico do tenant
   - Classificador heurístico por cenário preservado (função `pass_if(text, toolCalls)`) com comentário ensinando como escrever matcher bom vs ruim

2. **Saída padrão mantida idêntica ao smoke-scar.mjs**: relatório inline + JSON salvo em `/tmp/smoke-<tenant>-<timestamp>.json`.

3. **Código genérico**: `grep -iE "plaka|roberta|scar|gudesigner|nuvemshop|julia|fun personalize"` no template retorna **zero matches fora de comentários didáticos** (comentários marcados `// EXEMPLO:` podem referenciar tenants reais, código executável não).

4. **Documentação inline**: arquivo tem, no mínimo, 30% de linhas em comentário (medida via `grep -c "^\s*//"`). Comentários explicam o QUE e o POR QUÊ, não só o COMO.

5. **Testável "virgem"**: rodar `node --env-file=.env scripts/smoke-template.mjs --tenant=2` (PLAKA) com as variantes de exemplo ativas responde 5/5 ou 4/5 (dependendo do comportamento real de Roberta na data). Meta: template não exige customização pra rodar ao menos 1 cenário.

6. **Header compatível** com padrão de outros scripts da pasta (ver `chat-tenant.mjs` — shebang, comentário de uso, lista de pré-requisitos).

## Dev Notes

### Por que 2 variantes e não 1 template mais genérico?

Variantes separadas reconhecem que cenários de smoke são intrinsecamente ligados ao perfil do tenant:
- Prompt-only (ex: Scar): cenários sobre tom, idioma, objeções, escalação de tool críticas
- Com KB (ex: PLAKA): cenários sobre hit/miss, fuzzy fallback, escalation quando KB vazia

Forçar um único template genérico vira abstrato demais — usuário não sabe por onde começar. 2 variantes dão ponto de partida concreto.

### `pass_if` bem-feito

A função de matcher é o que diferencia smoke útil de smoke barulhento. Template DEVE ensinar via comentário:

```javascript
// ✅ BOM: heurística clara + alternativas
pass_if: (text) => /\b(Pix|pix)\b/.test(text) && !/\b(boleto)\b/.test(text)

// ❌ RUIM: substring match frágil
pass_if: (text) => text.includes('pix')
```

### Cenários críticos (D7 estilo)

Cenários que testam side-effect obrigatório (ex: invocou tool de escalação) DEVEM usar `critical: true` + matcher que verifica `toolCalls` (array de tool calls que rodaram). Isso é o aprendizado #1 do brief PLAKA — LLM simula tool sem executar. Template ensina pattern.

### Output determinístico

JSON em `/tmp/smoke-<tenant>-<timestamp>.json` continua — útil pra comparar runs.
