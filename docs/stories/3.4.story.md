# Story 3.4 — Mecanismo de Aplicação de Insights

## User Story
Como Cérebro Coletivo, quero um mecanismo para registrar a aplicação concreta de um insight validado com prova de melhoria mensurável, para que o ciclo obrigatório captura → validação → aplicação (FR6) seja completo e rastreável.

## Acceptance Criteria

- [x] 1. `PATCH /brain/insights/{id}/apply` implementado: aceita `applicationProof` (obrigatório), transiciona `validated → applied`, preenche `appliedAt` com timestamp atual. Retorna 400 se insight não estiver em status `validated`. Retorna 404 se insight não existir.
- [x] 2. `applicationProof` é validado: campos obrigatórios `appliedAt`, `appliedBy`, `changeDescription`, `baselineMetric`, `resultMetric`, `improvementPercent` devem estar presentes e tipados corretamente. Campos opcionais `storyId`, `nucleusId`, `evidenceRef` aceitos sem validação de conteúdo.
- [x] 3. `baselineMetric` e `resultMetric` seguem o schema `Metric` — campos obrigatórios: `name` (string), `value` (number), `unit` (string), `measuredAt` (string ISO 8601). `improvementPercent` deve ser um número (pode ser negativo — melhora nem sempre é positiva).
- [x] 4. `applicationProof` é persistido como JSONB no campo `application_proof` da tabela `insights` — recuperável via `GET /brain/insights/{id}` no campo `applicationProof`.
- [x] 5. `GET /brain/insights/{id}` já existente retorna `applicationProof` deserializado corretamente (não como string JSON) quando insight está `applied`.
- [x] 6. Testes unitários cobrindo: apply válido (`validated → applied`), apply com insight `raw` (400), apply com insight `rejected` (400), apply sem `applicationProof` (400), apply com `Metric` incompleto (400), `GET` retornando `applicationProof` deserializado.
- [x] 7. SOP criado em `docs/sops/sop-aplicar-insight.md` — processo para aplicar um insight validado, incluindo como calcular `improvementPercent`, quando usar `evidenceRef`, e como referenciar o `storyId`.

## Complexidade

## Riscos

- **Deserialização de JSONB:** `application_proof` é JSONB no Postgres — a camada db deve retornar como objeto, não como string. Verificar comportamento atual do `pg` ao fazer SELECT.
- **`improvementPercent` negativo:** Nem toda aplicação resulta em melhoria positiva — validar apenas que é `number`, não que seja positivo.
- **Transição de status estrita:** Apenas `validated → applied` é permitido. Não permitir `raw → applied` nem `rejected → applied`. Verificar status ANTES de persistir.

### Validação ApplicationProof

```typescript
function validateApplicationProof(proof: unknown): proof is ApplicationProof {
  if (!proof || typeof proof !== 'object') return false;
  const p = proof as Record<string, unknown>;
  
  // Obrigatórios
  if (typeof p.appliedAt !== 'string') return false;
  if (typeof p.appliedBy !== 'string') return false;
  if (typeof p.changeDescription !== 'string') return false;
  if (typeof p.improvementPercent !== 'number') return false;
  if (!validateMetric(p.baselineMetric)) return false;
  if (!validateMetric(p.resultMetric)) return false;
  
  return true;
}

function validateMetric(m: unknown): boolean {
  if (!m || typeof m !== 'object') return false;
  const metric = m as Record<string, unknown>;
  return (
    typeof metric.name === 'string' &&
    typeof metric.value === 'number' &&
    typeof metric.unit === 'string' &&
    typeof metric.measuredAt === 'string'
  );
}
```

O SOP deve cobrir:

- Como coletar métricas `resultMetric` após aplicar (aguardar período de estabilização)
- Como calcular `improvementPercent`: `((result - baseline) / |baseline|) * 100`
- Quando preencher `storyId` (quando a aplicação veio de uma story) e `evidenceRef`
