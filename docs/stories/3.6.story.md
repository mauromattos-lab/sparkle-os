## Riscos

- **Conflito de rota:** `POST /brain/insights/ingest` pode conflitar com `GET /brain/insights/{id}` se o roteador Hono não ordenar rotas corretamente. Verificar que `/ingest` é registrado antes da rota de parâmetro dinâmico `/:id`. Mitigação: registrar rota fixa antes de rota dinâmica no arquivo de rotas.
- **`confidenceLevel` override silencioso:** Sobrescrever o valor enviado pelo cliente para `mauro_input` pode surpreender integradores. Decisão intencional per `cerebro-coletivo.md §3.3` — documentar explicitamente no SOP e retornar o valor efetivo no response (não o enviado).
- **CLI em ambiente Windows:** `aiox brain add` deve funcionar no shell bash do ambiente de desenvolvimento (Windows 11 + bash). Verificar que o script Node.js usa shebang correto e que a passagem de flags funciona via `process.argv`.
- **sourceRef obrigatório para agent_research:** Impõe disciplina de rastreabilidade. Agentes precisam ser informados — coberto pelo SOP (AC 4).

## Dev Notes

### Arquitetura do Novo Endpoint

O novo endpoint `POST /brain/insights/ingest` é um thin wrapper sobre o pipeline existente. Não cria nova lógica de persistência nem de embedding — apenas aplica regras de validação e defaults antes de delegar.

```
POST /brain/insights/ingest
  │
  ├─ Validar: content presente, não vazio, <= 2000 chars → 400 se falhar
  ├─ Validar: source é 'mauro_input' | 'agent_research' → 400 se outro valor
  ├─ Se source='agent_research': sourceRef obrigatório → 400 se ausente
  ├─ Aplicar confidenceLevel: mauro_input → 'authoritative', agent_research → 'medium'
  └─ Delegar para ingest.service.ts (mesmo fluxo do POST /brain/insights)
      └─ gera embedding, canonicalização, persiste com status='raw'
```

**Fonte:** `docs/architecture/cerebro-coletivo.md §3.3` — tabela de confiança por fonte.

### Rota no Hono

```typescript
// packages/brain/src/routes/insights.ts
// IMPORTANTE: registrar ANTES de app.get('/brain/insights/:id', ...)
app.post('/brain/insights/ingest', async (c) => {
  const body = await c.req.json();
  
  // Validações source-specific
  if (!['mauro_input', 'agent_research'].includes(body.source)) {
    return c.json({ error: 'source must be mauro_input or agent_research' }, 400);
  }
  if (body.source === 'agent_research' && !body.sourceRef) {
    return c.json({ error: 'sourceRef is required for agent_research' }, 400);
  }
  
  // Override confidenceLevel — não confia no cliente
  const confidenceLevel = body.source === 'mauro_input' ? 'authoritative' : 'medium';
  
  // Delegar para o mesmo service
  const insight = await ingestService.ingest({ ...body, confidenceLevel });
  return c.json(insight, 201);
});
```

### Uso por Agentes (BrainClient)

Agentes usam o `BrainClient.ingest()` já disponível via `@sparkle-os/brain-client` (Story 3.5). ### Architectural Notes

1. **Padrão de thin wrapper bem executado:** O `/ingest` não duplica lógica — toda a persistência, embedding e canonicalização continuam no `ingest.service.ts`. Este padrão deve ser seguido em futuras extensões do endpoint base.

2. **`confidenceLevel` como propriedade do service, não do cliente:** A decisão de não expor `confidenceLevel` em `InsightInput` é arquiteturalmente sólida. O conhecimento de qual nível de confiança aplicar por fonte pertence ao sistema, não ao caller. Isso evita que callers erroneamente inflem a confiança de seus dados.

3. **BrainClient.ingest() — endpoint base vs /ingest:** O client continua chamando `POST /brain/insights`. Os testes do AC3 verificam o contrato do servidor via mock, não o URL. Para Story 3.7+, considerar se vale direcionar o client para `/ingest` explicitamente, dado que os agentes são os consumidores primários deste método.

4. **CLI como ponto de entrada para Mauro:** A escolha de `aiox brain add` como comando top-level no `bin/aiox.js` é extensível — abre o padrão para `aiox brain search`, `aiox brain list`, etc. em stories futuras sem refatoração.

5. **Validação de content no /ingest é mais rigorosa que no endpoint base:** O `/ingest` valida `null`, `undefined`, string whitespace e comprimento separadamente. O endpoint base valida apenas se é `string`. Esta assimetria é intencional (endpoint de conveniência com guardrails mais fortes) mas vale documentar em ADR se a distinção se mantiver em v2.
