## Story

**As** operador da Zenya com 7 tenants em produção,
**I want** mensagens entrantes nunca fiquem em status `pending` órfão na `zenya_queue`,
**so that** monitoring e operação fiquem confiáveis (status `pending` = realmente em processamento, não vazamento).

---

## Acceptance Criteria

1. **Fix 1 (Causa A — tenant lookup failure):** `webhook.ts` valida tenant via `loadTenantByAccountId(accountId)` ANTES de `enqueue`. Se falhar, retorna `400 unknown_tenant` SEM enfileirar
2. **Fix 2 (Causa B — test-mode skip):** linha 237-238 (test-mode skip path) chama `await markAllDone(pendingIds)` antes do `return`. Decisão semântica: `done` (não `failed`) — webhook fez seu trabalho corretamente decidindo ignorar
3. **Fix 3 (Causa C — failure path antes do try/catch interno):** TODA lógica dentro de `withSessionLock` envelope em try/finally robusto:
   - `pendingIds` declarado FORA do try
   - Catch best-effort: `await markAllFailed(pendingIds).catch(() => {})` — se DB tá fora do ar, nada a fazer
   - Re-throw após mark
4. **Fix 4 (Causa D — race condition):** após `markAllDone`, fazer `re-fetch` de `fetchPending`. Se houver mensagens novas que chegaram durante processamento, log warning + deixar pra próximo webhook pegar (não processar 2x no mesmo turn)
5. **Fix 5 — Cleanup script:** novo `packages/zenya/scripts/cleanup-stale-queue.mjs` marca pending velhos (>24h) como `failed`. Executar 1x após deploy pra zerar 875 atuais
6. Smoke 5 cenários (DB Specialist Review §3):
   - Tenant inexistente → 400 + zero pending
   - Test-mode tenant + phone fora whitelist → `done` (não pending)
   - Whisper falha forçada (mock) → `failed`, lock liberado
   - Chatwoot 5xx forçada → `failed`
   - Burst 5 mensagens em 30s → 1 ou 2 ciclos processam, **zero pending residual**
7. Após deploy + cleanup: `SELECT COUNT(*) FROM zenya_queue WHERE status='pending'` < 50 (mensagens recentes apenas)
8. Monitor 48h: pending count permanece estável <50 (sem crescimento monotônico)

## Dev Notes

**Estado pré-fix:** 875 mensagens pending acumuladas em 9 dias com 4 causas distintas:

| Causa | % | Mecanismo |
|-------|---|-----------|
| A — tenant lookup failure | 581 (66%) | Account 4 (Ensinaja) — webhook sem tenant; lookup lança; erro escapa |
| B — test-mode-skip leak | 34 (4%) | Account 7 (Scar) — `return` sem `markAllDone` |
| C — failure path antes try/catch interno | 221 (25%) | Julia — exceptions em fetchPending/Whisper/etc. escapam |
| D — race condition | 38 (4%) | Julia — msg nova entre fetch e markDone |

**Decisão semântica do `done` vs `failed` no test-mode-skip:** `done` (não `failed`). Razão: webhook fez seu trabalho corretamente (decidiu ignorar por test mode). Distinção importante pro alarme não sinalizar falsa instabilidade.

**Decisão sobre re-fetch (Fix 4):** mensagens novas chegadas durante processamento ficam pra próximo turn. Custo: pequena latência adicional (debounce 2.5s). Benefício: simplicidade + previsibilidade. Sem reentrancy.

**Cleanup script é one-shot:** executa 1x após deploy pra limpar histórico. Após Fix 1-4, leak ativo é zero — não precisa cron.
