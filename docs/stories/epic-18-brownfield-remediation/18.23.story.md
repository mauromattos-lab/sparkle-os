## Story

**As** dono de qualquer tenant Zenya recebendo leads via Click-to-WhatsApp ad (Instagram/Facebook),
**I want** o bot responder ao primeiro `incoming` do cliente mesmo que o WhatsApp Business app tenha disparado uma "Mensagem de saudação automática" `outgoing` antes,
**so that** leads de tráfego pago não morram em silêncio porque a saudação automática do app foi confundida com resposta humana e ativou `agente-off` antes do cliente falar.

## Acceptance Criteria

1. Em `packages/zenya/src/worker/webhook.ts:184-206`, antes de chamar `escalateToHuman({ source: 'human-reply' })`, validar que **já houve pelo menos um `incoming` do cliente nessa conversa**.
2. Quando a mensagem é `outgoing` em conversa que ainda não tem `incoming`, **NÃO aplicar `agente-off`** e responder com `{ ok: true, skipped: true, reason: 'outgoing_before_first_incoming' }`.
3. Sinal a usar (decisão `@architect` Aria — registrar em Dev Notes):
   - **Opção A (preferida se disponível):** `payload.conversation.messages_count` — Chatwoot v3 envia esse campo no webhook. Validar com amostra real antes de codar.
   - **Opção B (fallback):** query no DB Chatwoot `SELECT COUNT(*) FROM messages WHERE conversation_id = X AND message_type = 0` (0 = incoming na enum Chatwoot). Custo: 1 query extra por evento outgoing.
   - **Opção C (combinada):** preferir A; se ausente no payload, cair pra B.
4. Regra original preservada: outgoing **com `incoming` prévio** continua aplicando `agente-off` (humano respondeu pelo painel após o cliente falar — comportamento que existe hoje).
5. Bot reply (`content_attributes.sent_by_zenya === true`) continua ignorada como hoje (não muda nada).
6. Log claro do skip pra auditoria: `[zenya] outgoing_before_first_incoming — conv=<id> account=<id>` quando a guarda dispara, retornando antes do escalate.
7. Smoke test cobre 4 cenários:
   - Saudação automática WA Business **antes** de incoming → NÃO aplica `agente-off`
   - Humano responde no painel Chatwoot **depois** do incoming → aplica `agente-off` (regra preservada)
   - Bot responde (`sent_by_zenya=true`) → ignora (regra preservada)
   - Cliente manda incoming após saudação automática prévia → bot processa normalmente (não silenciado)
8. Sem regressão: testes existentes em `packages/zenya/src/__tests__/webhook.test.ts` continuam verde + novos cenários cobrindo AC 7.

## Dev Notes

### Cicatriz original

2026-04-26 — Gustavo (Scar AI) reportou via prints WhatsApp 14:46 que 2 leads vindos de anúncio IG ficaram sem resposta do bot:
- `+55 13 99177-0976` às 11:26 — *"Tenho interesse em deixar minha live mais profissional (VOL01)"*
- `+55 75 99956-4923` (Lucas.T) às 12:12 — mesma mensagem do anúncio (VOL01)

Em ambos, antes do incoming do cliente, o WhatsApp Business app do Gustavo enviou a "Mensagem de saudação automática" *"Oi! Como podemos ajudar?"*. Essa saudação:
- Sai como `message_type='outgoing'` no Chatwoot
- Não tem `content_attributes.sent_by_zenya === true` (não foi a Scar)
- Webhook trata como humano respondendo → aplica `agente-off`
- Quando cliente responde de fato, bot encontra label `agente-off` (linha 209) e silencia

### Análise da causa

Código atual em `packages/zenya/src/worker/webhook.ts:184-206`:

```ts
if (payload.message_type === 'outgoing') {
  const isBot = payload.content_attributes?.sent_by_zenya === true;
  const isHumanReply = !isBot;
  if (isHumanReply && !labels.includes('agente-off')) {
    await escalateToHuman({ source: 'human-reply' }); // ← aplica agente-off
  }
  return c.json({ ok: true, skipped: true });
}
```

Premissa errada da regra: assume que toda `outgoing` não-bot é resposta de humano via painel Chatwoot. Mas existem 3 origens de `outgoing` não-bot:
1. **Humano responde pelo Chatwoot panel** — `agente-off` faz sentido (parar bot)
2. **Humano responde pelo WhatsApp do número da loja** — `agente-off` faz sentido
3. **Saudação automática do WhatsApp Business app** — `agente-off` NÃO faz sentido (o cliente nem falou ainda)

Origem (3) é dominante em fluxo de Click-to-WhatsApp ad (Meta/IG/FB) — a saudação automática é configurada pelo dono do número e dispara antes do cliente.

### Por que essa story é cross-tenant blocker

Bug é determinístico para qualquer tenant Zenya:
- Que receba leads de Click-to-WhatsApp ads (IG/FB Meta)
- Que tenha "Mensagem de saudação automática" configurada no WhatsApp Business app

Tenants atualmente afetados: Scar (confirmado por Gustavo). Plaka, Fun, Doceria — risco se ativarem saudação automática. Cualquer tenant novo no SparkleOS — risco no momento que o dono ligar a saudação no app dele.

### Decisão técnica pendente — `@architect` Aria

Antes de codar, validar:

1. **Chatwoot v3 webhook envia `conversation.messages_count`?** Verificar amostra real dos webhook payloads (pm2 logs zenya-webhook salva ?), ou checar docs Chatwoot v3 (`https://www.chatwoot.com/developers/api/`).
2. Se sim → Opção A (mais barata, sem round-trip ao DB).
3. Se não → Opção B (query DB Chatwoot direto via psql ou via Chatwoot API `/api/v2/accounts/{id}/conversations/{id}/messages`).
4. Combinar (Opção C) se quiser robustez extra.

A guarda também serve como hardening genérico — mesmo que o tenant **não** tenha saudação automática, qualquer outgoing antes do primeiro incoming é caso degenerado que não deveria silenciar o bot.

## Risks

| Risco | Mitigação |
|-------|-----------|
| Chatwoot v3 não enviar `messages_count` no payload outgoing | Opção B (query DB) como fallback. Custo: 1 query extra por outgoing — tolerável |
| Query DB Chatwoot exige credenciais/permissões adicionais no zenya-webhook | Validar acesso na investigação `@architect`. Se restrito, usar Chatwoot API REST com token do tenant |
| Race condition: outgoing chega no webhook antes do incoming ser indexado no DB | Pouco provável (Chatwoot indexa síncrono). Se ocorrer, fallback é o comportamento atual (aplica agente-off) — mesmo bug do reporte |
| Tenant que NÃO tem saudação automática mas usa script externo enviando outgoing antes do cliente | Regra cobre genericamente: qualquer outgoing antes de incoming é skip. Não regride |
