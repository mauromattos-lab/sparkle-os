## Story

**As** dono operacional dos clientes Zenya,
**I want** o engine forçar a tool `escalarHumano` quando o LLM **promete** transferência no texto sem invocar a tool,
**so that** clientes não fiquem esperando atendente que nunca chega — comportamento crítico não pode depender 100% da decisão probabilística do LLM.

**Cicatriz original (cross-tenant ~20% flakiness):** observado em Julia (Prime), Roberta (PLAKA), Gustavo (Scar), HL durante refino. Iteração de prompt v1→v4.3 com checklist binário ("SE prometer THEN INVOCAR") atingiu ceiling em ~75% reliability. Conclusão QA gate HL-01 2026-04-22: **prompt iteration ceiling — próximo salto exige enforcement programático**.

## Acceptance Criteria

### AC1 — Detecção textual de promessa de handoff
Engine extrai `assistantMessage` antes de despachar pro Chatwoot. Conteúdo classificado como `intent: handoff_promised` quando matcheia padrões PT-BR (extensível por tenant):
- `"vou (te )?(transferir|encaminhar|passar) (pra|para) (um|uma) (atendente|humano|pessoa|colega)"`
- `"um (atendente|humano) (vai|irá|entrará em contato|te ajudar[áa])"`
- `"em breve (um|uma) (atendente|humano)"`
- (lista completa em código com testes por padrão)

### AC2 — Enforcement: promessa detectada + tool não invocada
Quando AC1 = `handoff_promised` E `escalarHumano` ausente em `tool_calls`:
1. **Invoca `escalarHumano` programaticamente** com defaults: `motivo='handoff_promised_in_text'`, `label='agente-off'`, `follow-up=true`
2. Loga `[guard] tenant=<slug> handoff_forced conversation=<id> reason=promised_without_invocation` (WARN)
3. Mensagem original ao cliente **mantida sem modificação** (cliente já recebeu a promessa; engine só garante side-effect)

### AC3 — Opt-out por tenant
Tenant com `guards.handoff_promise_enforcement: false` na config: AC1 dispara só log informativo, AC2 não atua. Default: `true` cross-tenant.

### AC4 — Cobertura de testes
- Detecção positiva: cada padrão default PT-BR resulta em `handoff_promised`
- Falso negativo: "atendente **já** te respondeu" NÃO dispara (passado, não promessa)
- Enforcement: promessa + tool ausente → `escalarHumano` invocada
- Opt-out: promessa + flag off → só log
- Idempotência: promessa + tool já presente → não invoca duas vezes
- Cross-tenant: smoke pré-deploy nos 4 ativos (Prime, PLAKA, Fun, HL/Doceria/Scar conforme disponibilidade)

**Mínimo: 15 testes; cobertura `guards/handoff-promise.ts` >= 90%**

### AC5 — Smoke cross-tenant pós-deploy
- Reliability `escalarHumano` invocada quando prometida: **>= 95%** (vs baseline ~75%)
- Zero regressão demais cenários (produtos, busca, agendamento)
- Logs `[guard] handoff_forced` aparecem em runs que antes falhariam

### AC6 — Documentação
- `docs/zenya/engine/guards/handoff-promise.md` — guard, padrões, opt-out, como adicionar padrão por tenant
- Cap. 1 §4 atualizado mencionando guards
- CHANGELOG marca breaking-safe (default = enforcement; tenants que dependiam da flakiness precisam opt-out explícito)

### AC7 — Rollback simples
Flag global env `ENGINE_GUARD_HANDOFF_PROMISE_ENABLED=true|false` desliga guard runtime. Default `true` em prod. Documentado em runbook.

## Dev Notes

**Threshold de detecção:** regex case-insensitive, opera em conteúdo *literal* da resposta do LLM (não em pensamento/raciocínio). Linha-base PT-BR; EN ficaria pra story futura se aparecer cicatriz.

**Por que software-side (não LLM-as-judge):**
- 1 segundo de overhead por turno NÃO aceitável (custo + latência)
- Regex O(n) determinística — <1ms benchmark esperado
- Defense in depth: prompt continua tentando, guard pega quando falha

**Não confundir com `escalarHumano` chamada normal:** o guard NÃO substitui a tool — apenas força quando texto promete sem chamada explícita. Fluxo normal LLM-decide-chamar continua igual.

**Referências canônicas:**
- Cap. 1 §4 (agent-loop)
- Frontend Spec §7 A6 (anti-pattern "promete e não faz")
- Memória `feedback_llm_simulates_tool` — same root cause (LLM textualiza ação sem executar)
- Gate HL-01 concern #1 waiver

## Risks

| Risco | Probabilidade | Mitigação |
|-------|---------------|-----------|
| Falso positivo: guard escala quando não deveria ("atendente já te respondeu no email") | M | Regex cuidadosa + teste explícito do caso + opt-out + flag global rollback |
| Regressão em tenant que dependia da flakiness | B | Opt-out + smoke cross-tenant pré-deploy |
| Performance regex em toda mensagem | B | <1ms benchmark esperado; O(n) tamanho msg |
| Conflito com prompt do tenant (prompt manda não prometer mas LLM promete) | B | É feature, não risco — guard resolve exatamente esse caso |
