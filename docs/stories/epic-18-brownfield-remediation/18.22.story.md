## Story

**As** dono operacional do SparkleOS adicionando cliente novo no core,
**I want** um único comando `aiox zenya:onboard <slug>` provisione tudo (seed Supabase + Chatwoot conta/labels/webhook + validação número + smoke gate automático),
**so that** onboarding de tenant novo leve **<30min de trabalho humano** ao invés das ~8h atuais (cópia-cola scripts + SQL ad-hoc + Chatwoot UI manual).

**Cicatriz original (2026-04-24 go-live Scar):** insight Mauro — *"se era pra termos já um padrão de colocar um cliente novo? isso me faz pensar que o padrão nao existe ainda"*. Após 7 tenants (Prime, Fun, HL, PLAKA, Doceria, Ensinaja, Scar), última milha continuou ad-hoc.

**Distinção crítica exposta:** Go-live (greenfield, ex: Scar nasceu no core) **≠** Cutover (brownfield n8n→core, ex: HL/Doceria). Template "cutover-checklist" foi clonado pra Scar erroneamente — não tem origem pra migrar.

---

## Acceptance Criteria

1. **CLI único:** `aiox zenya:onboard <slug> --mode=greenfield|cutover --from=n8n` recebe descriptor YAML e provisiona tenant ponta-a-ponta
2. **Modo greenfield:** prompt scaffold + Chatwoot conta+labels+webhook + upsert tenant Supabase com `allowed_phones` + gera `go-live-checklist.md`
3. **Modo cutover (n8n→core):** itens 2 + extração de prompt do workflow n8n + `cutover-runbook.md` com plano janela + rollback webhook
4. **Validação de número:** rejeita formato inválido (DDI obrigatório, dígitos consistentes), normaliza com/sem `9`, confirma interativamente em ambiguidade. Reusa `normalizePhone` de Story 18.13 (libphonenumber-js)
5. **Allowed_phones via CLI:** `aiox zenya:allow <slug> <phone>` faz UPDATE + invalida cache cross-instance (depende Story 18.9 NOTIFY/LISTEN)
6. **Idempotência:** rodar 2× com mesmo input não duplica nem corrompe estado
7. **Smoke gate automático:** após seed, roda `smoke-{slug}.mjs` (ou template canônico Story 18.10) — bloqueia "ready for go-live" se algum cenário falhar
8. **Migração legada:** 7 seed scripts (`seed-prime-tenant.mjs`, etc.) marcados deprecated mas continuam funcionando até último tenant migrar pro pipeline novo (depende Story 18.7)

## Dev Notes

**Esta story é meta-orchestrator** — depende de várias outras pra ficar completa:

| Sub-capacidade | Story que entrega |
|----------------|-------------------|
| Seed canônico unificado | Story 18.7 (Wave 2) |
| Cache invalidation cross-instance | Story 18.9 (Wave 2) |
| Smoke template canônico yaml | Story 18.10 (Wave 2) |
| normalizePhone com libphonenumber | Story 18.13 (Wave 2) |
| Onboarding orchestrator (esta) | Story 18.22 (Wave 2) |

**Sequenciamento Wave 2:** 18.7 + 18.9 + 18.13 antes de 18.22. 18.10 pode rodar em paralelo. 18.22 fecha a Wave 2 unificando tudo.

**Por que CLI (não UI):** UI gráfica é overkill na fase atual (Mauro = único operador). CLI cobre 100% dos casos com 10% do esforço. UI vira story futura se aparecer múltiplos operadores.

**Distinção go-live vs cutover preservada:** mesmo CLI, modos diferentes — gera output diferente, smoke diferente, rollback procedure diferente. **Não inventar fluxo unificado que confunde os 2.**

**Métrica de sucesso explícita:**
- **Antes:** ~8h por tenant greenfield (Scar = 19 tasks ao longo de 4 dias)
- **Depois:** **<30min** trabalho humano (CLI + QR pareamento Z-API + smoke)

**Fora do escopo (preservar):**
- UI gráfica (CLI suficiente)
- Provisionamento Z-API (continua manual — depende QR cliente)
- Treinamento cliente final (humano)
- Onboarding multi-canal Instagram/Telegram (out)

**Referências canônicas:**
- DB Specialist Review TD-09 (seed unification)
- Cap. 3 §3 (onboarding novo cliente — fluxo atual ad-hoc)
- Cap. 8a Template Canônico §4 (cross-nucleus pattern)
- Memória `feedback_automation_over_input` P9: AI infere do contexto, não pede input redundante

## Testing

**Smoke local:**
- Criar descriptor YAML fake `test-tenant.yaml` greenfield
- Rodar `aiox zenya:onboard test-tenant --mode=greenfield --dry-run` → output simulado completo
- Rodar real em tenant fake → validar:
  - Chatwoot conta criada (ou skip se já existe)
  - Tenant upsertado em `zenya_tenants`
  - Smoke gate roda + reporta
- Cleanup: `aiox zenya:remove test-tenant` (não-destrutivo se opt-out)

**Smoke produção:**
- Próximo cliente real entra via pipeline novo
- Cronômetro: <30min do `aiox zenya:onboard ... ` até "ready for go-live"
- Rollback: testar com tenant fake — `aiox zenya:remove` + verificar zenya_tenants limpo

**Critério de saída:** AC 1-8 atendidos + smoke real com cliente novo OK.
