## Story

**As** cliente em modo de teste (allowed_phones com número admin) cuja conversa foi silenciada pelo label `agente-off`,
**I want** o comando `/reset` também remover o label `agente-off` da conversation no Chatwoot,
**so that** eu consigo destravar o bot sozinho via WhatsApp sem precisar do dono abrir Chatwoot manualmente.

**Cicatriz original (2026-04-24):** durante go-live do Scar AI, Gustavo (cliente) ficou travado após `agente-off` aplicado por escalação. `/reset` existia mas só limpava `clearHistory`, não tocava no label. Cleanup automático só roda 72h. Mauro precisou abrir Chatwoot e remover label manualmente — fricção inaceitável no fluxo de validação cliente.

## Acceptance Criteria

1. **Quando `config.allowed_phones.length > 0` AND `mergedMessage.trim() === '/reset'`:**
   - Mantém: `clearHistory(config.id, phone)`
   - **Novo:** `removeConversationLabel(chatwootParams, 'agente-off')` — idempotente
   - Mensagem ajustada conforme outcome (3 variantes — ver AC4)
2. **Em modo produção (`allowed_phones.length === 0`):** comportamento inalterado — `/reset` não tem tratamento especial (evita porta de abuse pra clientes em prod)
3. **`removeConversationLabel`:** função nova em `packages/zenya/src/integrations/chatwoot.ts`, simétrica à `addLabel`:
   - GET `/labels` (lista atual)
   - Filter out target label
   - POST replace (Chatwoot API é replace, não delta)
   - Idempotente: retorna `{ removed: false, reason: 'not-present' }` quando label já estava ausente — skip POST
4. **Mensagens de confirmação (3 variantes):**
   - `removed=true` → `"🔄 Memória zerada + bot reativado. Nova conversa!"`
   - `removed=false, reason=not-present` → `"🔄 Memória zerada. Nova conversa!"`
   - `removed=false, reason=fetch_failed|throw` → `"🔄 Memória zerada. Se o bot não responder, peça pro admin remover 'agente-off'."` (fallback graceful)
5. **Log estruturado:** `[zenya] /reset — tenant=${config.id} phone=${phone} cleared_history=true removed_label=${result.removed} reason=${result.reason ?? 'removed'}`
6. **Testes unitários** em `webhook.test.ts` cobrindo:
   - Test mode com label `agente-off` presente → `clearHistory` + `removeConversationLabel` chamados, mensagem variant 1
   - Test mode SEM label → `clearHistory` chamado, `removeConversationLabel` chamado idempotente, mensagem variant 2
   - Test mode com Chatwoot API falhando → fallback variant 3, conversa não trava
   - Modo produção (`allowed_phones=[]`) → nenhum tratamento especial (fluxo agente normal)
7. **Sem regressão:** suite `npm test` continua passando 100% (atualmente 111/111 pós-Story 18.1; +4 testes novos = 115/115)

## Dev Notes

**Decisão técnica (memória `feedback_agent_decides_technical`):**
- `removeConversationLabel` segue padrão simétrico de `addLabel` existente em `chatwoot.ts:70`
- Chatwoot API é replace (POST com array completo) — não delta — reaproveitar pattern
- Idempotência via early-return evita POST desnecessário (latência + risco)
- Fallback graceful (variant 3) atende princípio P9: nunca deixar conversa travada quando podemos comunicar opção alternativa

**Por que NÃO em modo produção (AC2):**
- Em prod, clientes reais não devem ter comando mágico de re-engajamento (porta de abuse)
- Se cliente em prod travar via `agente-off`, fluxo correto é cleanup 72h ou escalação humana
- Story futura (post-launch) pode adicionar rate-limited `/reset` em prod com gate de owner

**Cuidado polissemia ChatwootParams:**
- `getChatwootParams(accountId, conversationId)` retorna struct usado pela `addLabel` — reaproveitar
- Não criar struct novo

**Referências canônicas:**
- Frontend Spec §7 A10 (anti-pattern label travada)
- Cap. 3 §10 (debugar bot silenciado)
- Cap. 6 OP9 (reset em test mode)

## Risks

| Risco | Mitigação |
|-------|-----------|
| Cliente em test mode abusa `/reset` em loop | Aceitável — test mode é efêmero. Em prod o comando não é tratado especial (AC2) |
| Chatwoot API 500 quando remove label | Fallback variant 3 (AC4) — usuário recebe mensagem informando alternativa |
| Conversa sem `agente-off` quando `/reset` chamado | AC3 idempotência — no-op silencioso, variant 2 |
| Bug novo impacta tenants em produção | Gate estrito `allowed_phones.length > 0` — código novo só executa em test mode |
