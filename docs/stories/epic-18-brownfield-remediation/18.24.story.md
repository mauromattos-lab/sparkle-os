## Story

**As** dono Zenya recebendo cliente em conversa por áudio (TTS),
**I want** a voz da Scar AI (e capacidade futura pra outros tenants) soar **natural, jovem, expressiva e curta** — combinando com o público-alvo dos streamers,

## Diagnóstico (`@architect` Aria, 2026-04-27)

Configuração TTS atual em `packages/zenya/src/integrations/elevenlabs.ts`:

| Parâmetro | Valor atual | Problema |
|-----------|-------------|----------|
| `model_id` | `eleven_flash_v2_5` | **Otimizado pra latência, não pra emoção.** Flash é monotônico por design — sempre vai soar robotizado. |
| `voice_settings` | `{stability: 0.5, similarity_boost: 0.75}` hardcoded cross-tenant | Sem param `style` (suportado por v2/v3). Stability 0.5 é "neutra" — pouca variação emocional. |

| Comprimento da resposta | Sem limite explícito no prompt | Respostas longas viram áudios longos e cansativos. Gustavo queixou-se de "longo". |

### Defaults vs override

A solução faz **2 mudanças simultâneas**:

1. **Defaults globais elevados** (cross-tenant): novos defaults hardcoded passam de `flash_v2_5/{0.5, 0.75}` para `multilingual_v2/{stability: 0.40, similarity_boost: 0.65, style: 0.40, use_speaker_boost: true}`. **TODOS os tenants** se beneficiam automaticamente da qualidade melhor.
2. **Override per-tenant via `tts_config`**: tenant pode setar voice_id customizado (Scar) e/ou ajustar settings. Coluna nullable — se NULL, usa defaults globais novos.

> **Trade-off documentado:** Zenya Prime (tenant do próprio Mauro, calibrado há semanas no flash_v2_5) **vai mudar de voz/timbre** automaticamente após esta migration. Mauro pode opt-out via `UPDATE zenya_tenants SET tts_config = '{"model_id":"eleven_flash_v2_5","voice_settings":{"stability":0.5,"similarity_boost":0.75}}' WHERE name='Zenya Prime (SparkleOS)'` se quiser preservar.

## Dev Notes

### Por que `eleven_multilingual_v2` e não `eleven_v3`

`eleven_v3` foi lançado recentemente em alpha pela ElevenLabs com expressividade ainda maior (suporta tags emocionais inline tipo `[laughs]`). Mas alpha = instabilidade. Pra um sistema de produção atendendo cliente pagante (Gustavo), o trade-off não compensa o ganho. `multilingual_v2` é production-stable há ~2 anos com qualidade comprovada e MUITO superior ao `flash_v2_5` em expressividade. Reavaliar v3 quando ElevenLabs marcar `production-ready`.

### Por que coluna JSONB e não colunas separadas

Padrão consistente com migration 009 (`audio_format` como TEXT) seria adicionar 3 colunas (`tts_voice_id`, `tts_model_id`, `tts_voice_settings`). Mas `voice_settings` é estrutura aninhada (4 fields nested) — virar JSONB é natural. Manter tudo em UMA coluna JSONB (`tts_config`) mantém o objeto coeso semanticamente: "configuração TTS opcional do tenant". Outro tenant que adicione futuro pode estender o JSONB sem migration nova (e.g. `pre_processing_ssml: false`).

### Padrão de fallback (cascata)

```typescript
function resolveTtsConfig(tenantConfig: TenantConfig): ResolvedTtsConfig {
  return {
    voice_id: tenantConfig.tts_config?.voice_id ?? process.env['ELEVENLABS_VOICE_ID'],
    model_id: tenantConfig.tts_config?.model_id ?? 'eleven_multilingual_v2',
    voice_settings: {
      stability: tenantConfig.tts_config?.voice_settings?.stability ?? 0.40,
      similarity_boost: tenantConfig.tts_config?.voice_settings?.similarity_boost ?? 0.65,
      style: tenantConfig.tts_config?.voice_settings?.style ?? 0.40,
      use_speaker_boost: tenantConfig.tts_config?.voice_settings?.use_speaker_boost ?? true,
    },
  };
}
```

### Compatibilidade com `audio_format` (migration 009)

`tts_config` (010) e `audio_format` (009) são ortogonais — `audio_format` decide o output container (mp3 vs ogg_opus, formato do arquivo), `tts_config` decide a voz/modelo/expressividade. Ambos podem coexistir num tenant. Scar usa `audio_format='mp3'` (Z-API) + `tts_config={voice_id, voice_settings}` (Story 18.24).

## Risks

| Risco | Mitigação |
|-------|-----------|
| `eleven_multilingual_v2` mais lento que flash → Scar leva >5s pra responder áudio | Latência v2 é 400-800ms, suíte WhatsApp aceita 5-15s. SLA mantido. Se medirmos >3s consistente, abrir story de cache TTS. |

| Migration falha em prod | Migration nullable + idempotente (`IF NOT EXISTS`). Rollback: `ALTER TABLE zenya_tenants DROP COLUMN tts_config`. Snapshot SQL prévio padrão. |
| Default novo cross-tenant degrada Zenya Prime (Mauro calibrou no flash) | Documentado em Dev Notes. Mauro pode opt-out preservando flash via `tts_config = {"model_id":"eleven_flash_v2_5", ...}`. Avaliar pós-deploy. |
