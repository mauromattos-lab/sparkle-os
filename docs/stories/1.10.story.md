## Technical Context

### Princípios de Segurança (de `docs/architecture.md` §2.5)
- **Multi-tenant by design:** `tenant_id` em todas as tabelas de dados de clientes
- **Row-Level Security (RLS):** Postgres RLS como mecanismo de isolamento — não na aplicação
- **Auth:** IP allowlist + token interno para API — sem OAuth (sistema interno, usuário único Mauro)
- **Zero trust para dados de clientes:** Nenhuma query cross-tenant possível por design

### Modelo de Tenant
```
Tenant = Cliente Zenya
  → Cada cliente tem tenant_id único (UUID)
  → Todas as conversas, dados e configurações isolados por tenant_id
  → Agentes acessam dados sempre com tenant context explícito
```

### Schema Base com RLS

```sql
-- Tabela de tenants (clientes Zenya)
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,        -- identificador humano (ex: 'empresa-abc')
  status TEXT NOT NULL DEFAULT 'active',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Exemplo: tabela com isolamento por tenant
CREATE TABLE tenant_data (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  -- ... campos específicos
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- RLS: habilitar
ALTER TABLE tenant_data ENABLE ROW LEVEL SECURITY;

-- Policy: agentes acessam apenas dados do tenant no contexto atual
CREATE POLICY tenant_isolation ON tenant_data
  USING (tenant_id = current_setting('app.current_tenant_id')::UUID);
```

### Modelo de Auth da API
```
SparkleOS Internal API (Hono):
  - Acesso: Bearer token estático (gerado na Story 1.5)
  - Sem JWT, sem OAuth — sistema interno
  - Token em variável de ambiente: INTERNAL_API_TOKEN

Supabase (direto):
  - Agentes e aplicação usam service_key para operações internas
  - Nunca expor service_key no frontend
  - anon_key apenas para operações públicas (nenhuma por ora)
```
