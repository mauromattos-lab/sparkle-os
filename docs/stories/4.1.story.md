# Story 4.1 — Dashboard do Sistema — Shell do Cockpit

## User Story

Como Mauro, quero uma visão geral do SparkleOS em tempo real, para que eu entenda o estado do sistema em segundos sem abrir logs ou perguntar para os agentes.

## Status
- [x] Draft
- [x] Ready
- [ ] InProgress
- [ ] InReview
- [ ] Done

## Acceptance Criteria

- [ ] 1. Shell do cockpit funcionando em `http://localhost:3004` — navegação entre painéis sem erro (Health, Agentes, Decisões, Zenya, Cérebro, Custos, Progresso, Resumo)
- [ ] 2. Saúde das integrações visível: Brain API (`/brain/health`), status do banco Supabase — indicadores claros (verde/vermelho) sem texto técnico
- [ ] 3. Alertas ativos destacados com contexto suficiente para Mauro entender sem ser técnico — mínimo: alertas de integração offline e stories bloqueadas
- [ ] 4. Atualização automática a cada 30 segundos (polling) sem necessidade de refresh manual
- [ ] 5. SOP inline: cada tipo de alerta tem texto explicando o que Mauro deve fazer — sem link externo, direto na tela

## Escopo

**IN (esta story entrega):**
- Package `packages/cockpit/` — novo package no monorepo pnpm
- Servidor HTTP na porta 3004 (Hono, padrão do projeto)
- Layout base com navegação lateral entre painéis (links, não SPAs)
- Painel `/cockpit/` — visão geral do sistema: saúde das integrações, alertas, status rápido
- Polling de 30s para atualizar dados sem reload
- `.env.example` com variáveis necessárias
- `package.json` com scripts `dev`, `build`, `test`

**OUT (não inclui):**
- Implementação dos painéis internos (Agentes, Zenya, Cérebro, Custos, Progresso, Resumo) — → Stories 4.2–4.8
- Autenticação / acesso externo — cockpit é local-only neste epic
- Deploy no Vercel — infra é escopo de Epic futuro
- Gráficos ou visualizações complexas — indicadores simples (texto + cor) são suficientes

## Complexidade
**M (Médio)** — Novo package com stack conhecida (Hono + HTML server-side, padrão de `packages/brain/`). Sem lógica de negócio complexa nesta story — foco em estrutura, navegação e saúde das integrações.

## Riscos

- **Decisão de stack:** A arquitetura menciona Next.js para a Piloting Interface (deploy no Vercel), mas o padrão atual do projeto usa Hono + HTML server-side (packages/brain). Para desenvolvimento local e este epic, Hono é mais consistente com o padrão existente. @architect deve confirmar ou registrar ADR se decidir diferente.
- **Porta em conflito:** Brain usa 3003. Cockpit usa 3004. Verificar que não há conflito.
- **Dados sem fonte real:** O painel de Overview desta story deve mostrar apenas dados que realmente existem agora (Brain health via API, git log). Sem dados inventados.

### API do Brain disponível
[Source: packages/brain/src/routes/health.ts — Story 3.x Done]

```
GET http://localhost:3003/brain/health
→ { status: "ok", db: "ok", embeddingService: "ok" }
```

O cockpit deve consumir este endpoint para mostrar o status do Brain no Overview.

### Padrão de HTML server-side (referência)
[Source: packages/brain/src/routes/dashboard-ui.ts — Story 3.9]

Usar template literal HTML igual ao padrão já estabelecido no Brain. Não usar framework de template externo.

```typescript
// Padrão do projeto — retornar HTML diretamente
app.get('/cockpit', (c) => {
  return c.html(`<!DOCTYPE html>...`)
})
```

### Variáveis de ambiente necessárias
```env
COCKPIT_PORT=3004
BRAIN_URL=http://localhost:3003  # URL do Brain para checagem de saúde
NODE_ENV=development
```

### Padrão de navegação
Links HTML simples entre rotas (`<a href="/cockpit/agents">`). Sem SPA, sem React. Consistente com dashboard-ui do Brain.
