# Story 2.3 — Integração Formal da Zenya como Núcleo

## User Story
Como agente AIOS, quero que a Zenya esteja formalmente registrada como Núcleo do SparkleOS — com contrato de inputs/outputs definido, Zenya Adapter funcional e API interna respondendo — para que qualquer agente possa interagir com a Zenya de forma programática e rastreável, sem precisar conhecer o n8n diretamente.

## Acceptance Criteria

- [ ] 1. `docs/zenya/NUCLEUS-CONTRACT.md` criado — define formalmente: o que a Zenya recebe (inputs), o que produz (outputs), qual é o seu SOP e como se encaixa na arquitetura de Órgãos do SparkleOS
- [ ] 2. `organs/zenya/src/` estruturado com código funcional — Zenya Adapter com Hono, cliente n8n e rotas base implementados; `npm run build` e `npm run typecheck` passando

### Zenya Adapter — Arquitetura
[Source: docs/architecture.md §6 — Zenya Adapter]
```
Responsabilidade: Integração com n8n + Chatwoot + Z-API
                  Encapsula toda complexidade — o resto não conhece n8n diretamente
Stack: Node.js 22 + Hono — Coolify na VPS
Acesso n8n: http://localhost:5678/api/v1 (mesma VPS — zero latência)
```

### Decisão Pendente: Topologia de Deploy
A ADR (AC6) deve resolver:
- **Opção A — Serviço independente:** Zenya Adapter em porta própria (3002), registrado no Core como Nucleus service
- **Opção B — Integrado ao Core:** Rotas `/nucleus/zenya/*` montadas diretamente no Core (packages/core)

Ambas as opções têm trade-offs. @architect decide e documenta na ADR antes de implementar.

[Source: docs/architecture.md §5 API Specification]
