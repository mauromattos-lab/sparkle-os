# Story 3.1 — Arquitetura do Cérebro Coletivo

## User Story
Como agente AIOS, quero a arquitetura do Cérebro Coletivo definida e documentada antes de qualquer implementação, para que o que construirmos suporte múltiplas fontes de conhecimento desde o design inicial e não precise ser refeito ao escalar para novos Núcleos e Órgãos.

## Acceptance Criteria

- [ ] 1. Modelo de dados do `Insight` expandido e finalizado a partir da definição inicial em `docs/architecture.md §4` — campos, tipos, relacionamentos, estratégia de indexação pgvector documentados com justificativa de cada decisão
- [ ] 2. Arquitetura de fontes documentada com contrato de como cada fonte alimenta o Cérebro: `zenya_operation` (Zenya Adapter), `agent_research` (agentes AIOS), `mauro_input` (Mauro via interface) — atributos de origem, confiança diferenciada e rastreabilidade definidos para cada fonte
- [ ] 3. Mecanismo de canonicalização definido: estratégia para detectar que dois insights representam o mesmo conceito e evitar fragmentação — deve ser pragmática (implementável em Story 3.2), não perfeita
- [ ] 4. ADR criado em `docs/adrs/` documentando: confirmação de pgvector como vector store, modelo de embeddings escolhido (Voyage via Anthropic SDK), threshold de busca semântica (0.75 como baseline), e qualquer desvio da arquitetura inicial
- [ ] 5. SOP criado em `docs/sops/` descrevendo como adicionar nova fonte de dados ao Cérebro no futuro — processo executável por agentes sem redesenho da arquitetura

## Riscos

- **Desvio da arquitetura inicial:** `docs/architecture.md` já define o modelo `Insight` e endpoints Brain — qualquer desvio significativo requer atualização do documento base e ADR justificando a mudança
- **Canonicalização superestimada:** estratégia de deduplicação semântica é problema difícil — solução deve ser pragmática e implementável, não ideal teórico
- **Threshold de busca inadequado:** 0.75 é o baseline definido na arquitetura — @architect deve confirmar ou ajustar com justificativa baseada nas características dos dados Zenya

### Contexto Arquitetural Existente (fonte: `docs/architecture.md`)

**Modelo `Insight` já definido (§4 — Data Models):**
```typescript
interface Insight {
  id: string;
  source: 'zenya_operation' | 'agent_research' | 'mauro_input';
  nucleusId: string | null;
  content: string;

  status: 'raw' | 'validated' | 'applied' | 'rejected';
  qualityScore: number | null;
  validationNotes: string | null;
  applicationProof: string | null;  // FR6: obrigatório para ciclo completo

  createdAt: string;

}
```

**Endpoints Brain já esboçados (§5 — API Specification):**
```
POST   /brain/insights                    # Ingestão — gera embedding automaticamente
POST   /brain/insights/search             # Busca semântica (threshold 0.75, só validated/applied)
PATCH  /brain/insights/{id}/validate      # raw → validated
PATCH  /brain/insights/{id}/apply         # registrar aplicação mensurável (FR6)
```

**Componente Collective Brain (§6 — Components):**
- Stack: Node.js 22 + Hono + pgvector — Coolify na VPS
- Separado do Core: requisitos distintos (embeddings, vetorial, ML pipeline)
- Location: `packages/brain/` (novo package no monorepo pnpm)
- pgvector via Supabase free tier — sem serviço adicional

**Padrão arquitetural (§2.5):**
```
Knowledge Pipeline: Ingestão → Validação → Aplicação (pipeline separado do operacional)
```

### Perguntas abertas para @architect resolver nesta story

1. **Dimensionalidade dos embeddings:** Voyage suporta 1024 e 1536 dims — confirmar qual usar considerando tradeoff custo/precisão
2. **Canonicalização:** similarity threshold para detecção de duplicata (ex: cosine similarity > 0.92?) ou clustering (ex: DBSCAN)? Ou abordagem mais simples para v1?
3. **Schema pgvector:** índice IVFFLAT ou HNSW? Para o volume esperado (centenas de insights no v1) HNSW é mais adequado — confirmar
4. **Confiança por fonte:** `zenya_operation` (alta — dados reais), `agent_research` (média — pode ter imprecisão), `mauro_input` (alta — autoridade máxima) — formalizar no modelo
5. **`applicationProof`:** string livre ou schema estruturado? Estruturado permite métricas automáticas
