# Story 7.9 — Debounce de Mensagens e Confirmação de Leitura

**Epic:** 7 — Zenya como Órgão Nativo do SparkleOS

## Descrição

**As a** usuário do WhatsApp,
**I want** que mensagens enviadas em sequência rápida sejam agrupadas em uma resposta só e que eu veja o "lido" imediatamente,
**so that** a Zenya responda de forma coerente mesmo quando envio várias mensagens seguidas e eu tenha confirmação visual de que fui atendido.

---

## Acceptance Criteria

- [x] AC1: Webhook aguarda `ZENYA_DEBOUNCE_MS` (padrão 2500ms) após adquirir o lock antes de processar — todas as mensagens que chegam durante esse intervalo são coletadas da fila e unificadas em um único input para o LLM
- [x] AC2: `fetchPending()` retorna todas as mensagens `status = 'pending'` do par `(tenant_id, phone_number)` ordenadas por `created_at ASC` — conteúdos unidos com `\n`
- [x] AC3: Ao final do processamento (sucesso ou falha), todos os `message_id` do burst são marcados com `markAllDone()` ou `markAllFailed()` — nunca ficam presos em `pending`
- [x] AC4: `markConversationRead()` chamado no início do processamento (antes do typing indicator) via `POST /conversations/{id}/update_last_seen` — usuário vê double-check imediatamente
- [x] AC5: Falha em `markConversationRead` é não-crítica — `console.warn` apenas, sem lançar exceção
- [x] AC6: `ZENYA_DEBOUNCE_MS` configurável via variável de ambiente — documentado em `.env.example`
- [x] AC7: Testes unitários cobrem: debounce merge de múltiplas mensagens, fallback para mensagem atual se `fetchPending` retorna vazio, `markAllDone` e `markAllFailed`

---

## Escopo

**IN:**
- Debounce window no webhook handler (sleep + fetchPending)
- Funções `fetchPending`, `markAllDone`, `markAllFailed` na fila
- Marcação de leitura via `markConversationRead` na integração Chatwoot
- Env var `ZENYA_DEBOUNCE_MS` documentada

**OUT:**
- Indicador de "gravando áudio" no WhatsApp — requer camada Z-API/Baileys, fora do escopo da Chatwoot API REST
- Merge inteligente de mensagens (ex: interpretar contexto entre elas) — processamento semântico é responsabilidade do LLM ao receber o texto unido

## Dev Notes

### Debounce — como funciona

O lock (`withSessionLock`) garante que apenas uma execução por sessão roda por vez. O debounce acontece *dentro* do lock:

```typescript
const DEBOUNCE_MS = parseInt(process.env['ZENYA_DEBOUNCE_MS'] ?? '2500', 10);

await withSessionLock(accountId, phone, async () => {
  // 1. Dormir para deixar mensagens do burst chegarem e serem enfileiradas
  await sleep(DEBOUNCE_MS);

  // 2. Coletar todas as pendentes (inclui a mensagem atual e qualquer burst)
  const pending = await fetchPending(accountId, phone);
  const mergedMessage = pending.length > 0
    ? pending.map((m) => m.content).filter(Boolean).join('\n')
    : message; // fallback: a mensagem que disparou o webhook

  try {
    await runZenyaAgent({ ..., message: mergedMessage });
    await markAllDone(pending.map(m => m.message_id));
  } catch (err) {
    await markAllFailed(pending.map(m => m.message_id));
    throw err;
  }
});
```

**Por que funciona:** A segunda mensagem do burst chega enquanto o lock está aguardando (sleep). Ela é enfileirada normalmente via `enqueue()`. Quando o sleep termina, `fetchPending()` encontra ambas as mensagens e as une.

### Indicador de leitura — ordem de operações

```typescript
// agent/index.ts — início do processamento
await markConversationRead(chatwootParams).catch(() => undefined); // double-check imediato
await setTypingStatus(chatwootParams, 'on').catch(() => undefined); // typing indicator
// ... LLM call ...
// finally:
await setTypingStatus(chatwootParams, 'off').catch(() => undefined);
```

**Nota:** A Chatwoot API REST não expõe estado de "gravando áudio". Para WhatsApp, isso exigiria acesso direto à camada Z-API/Baileys que o Chatwoot encapsula. A Zenya usa apenas `typing on/off`.

### Variáveis de ambiente relevantes

```bash
ZENYA_DEBOUNCE_MS=2500  # janela de espera em ms (padrão: 2500)
```
