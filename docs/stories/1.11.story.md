## Technical Context

### Cenário de Validação End-to-End

```
Sessão A (@sm):
1. Criar story de validação no SparkleOS via *draft
2. Salvar contexto via Context Store API
3. Registrar ADR de decisão de teste via ADR API
4. Encerrar sessão

Sessão B (@dev):
5. Recuperar contexto da Sessão A sem reexplicar
6. Marcar story como InProgress
7. Consultar ADR criada pelo @sm
8. Implementar mini-feature de validação (placeholder)
9. Registrar custo da sessão

Sessão C (@qa):
10. Executar QA Gate na story de validação
11. Dar PASS formal
12. Verificar que custo total das 3 sessões está rastreado
13. Marcar story Done
```

### Critérios de Passagem por AC
| AC | Como validar | Evidência |
|----|-------------|-----------|
| 1 | Story criada em `docs/stories/` pelo processo AIOS | Arquivo `.story.md` existe |
| 2 | Story tem status `Done` com checkboxes do lifecycle marcados | Arquivo atualizado |
| 3 | `GET /api/context/:agentId` retorna contexto da sessão anterior | Response da API com `workState` populado |
| 4 | `GET /api/adrs/:number` acessível em nova sessão | Response com ADR completa |
| 5 | `GET /api/costs/summary?period=daily` mostra eventos da sessão | Response com `totalCost > 0` |
