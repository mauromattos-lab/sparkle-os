# Story 3.9 — Dashboard do Cérebro

## User Story

Como operador do SparkleOS (Mauro) ou agente AIOX, quero um dashboard com dados agregados do Cérebro Coletivo, para que eu possa visualizar o estado do conhecimento acumulado — insights por status, por fonte, métricas do ciclo completo (captura → validação → aplicação), quality scores, duplicatas detectadas e melhorias mensuráveis aplicadas — sem precisar consultar a base de dados diretamente.

## Acceptance Criteria

1. **Endpoint `GET /brain/dashboard`** implementado no `packages/brain/` e exposto via gateway `/brain/dashboard`. Retorna JSON com estrutura `DashboardData` conforme especificado nos Dev Notes. Resposta em < 500ms (cache em memória com TTL 60s aceito).

2. **Seção `summary`** do payload inclui: total de insights, contagem por status (`raw`, `validated`, `applied`, `rejected`), contagem por fonte (`zenya_operation`, `agent_research`, `mauro_input`), total de duplicatas detectadas (`isDuplicate: true`), média de `qualityScore` dos insights validados/aplicados (2 casas decimais).

3. **Seção `cycle`** do payload representa o funil do ciclo obrigatório (FR6): `ingested` (total `raw` criados), `validated` (total que chegaram a `validated` ou além), `applied` (total com `status=applied`), `rejected` (total `rejected`), e `completionRate` (percentual `applied / ingested`, 2 casas decimais).

4. **Seção `insights_by_confidence`** agrupa contagens por `confidenceLevel` (`authoritative`, `high`, `medium`) — apenas insights não-rejeitados.

5. **Seção `top_applied`** retorna os 5 insights mais recentes com `status=applied`, cada um incluindo: `id`, `summary`, `source`, `confidenceLevel`, `appliedAt`, `improvementPercent` (extraído de `applicationProof.improvementPercent`) e `nucleusId`. Campo `improvementPercent` é `null` se `applicationProof` ausente.

6. **Seção `quality_distribution`** retorna histograma de `qualityScore` em 5 buckets: `0.0–0.2`, `0.2–0.4`, `0.4–0.6`, `0.6–0.8`, `0.8–1.0` — contagem de insights validados ou aplicados em cada faixa.

7. **Seção `duplicates`** inclui: total de insights marcados como duplicatas (`isDuplicate: true`), lista dos 5 insights canônicos com mais duplicatas apontando para eles (campo `canonical_id`), cada item com `canonicalId`, `count`, `summary` do canonical.

8. **Página HTML estática** servida em `GET /brain/dashboard/ui` — renderiza os dados de `GET /brain/dashboard` em formato legível via `fetch` client-side. Deve exibir: summary cards (totais), tabela de ciclo com completion rate destacado, lista top applied com improvement%, e tabela de distribuição de qualidade. Sem frameworks externos — HTML + CSS inline + JS vanilla. Aceito como "simples e funcional" para viabilizar Epic 4 partir daqui.

9. **Testes de integração** em `packages/brain/src/dashboard.test.ts` cobrindo: resposta 200 com estrutura `DashboardData` completa, `summary.total` reflete inserções reais, `cycle.completionRate` calculado corretamente com dados seed, `top_applied` limitado a 5 e ordenado por `appliedAt` desc, endpoint `/brain/dashboard/ui` retorna HTML (status 200, `Content-Type: text/html`).

10. **SOP criado** em `docs/sops/sop-dashboard-cerebro.md` — como acessar o dashboard (URL interno e via gateway), como interpretar cada seção, quando usar o endpoint JSON vs. a página HTML, e nota de coordenação com Epic 4 (o endpoint JSON é o contrato de dados que o frontend do Epic 4 consumirá).

## Complexidade

**M (Médio)** — Novas queries de agregação SQL (GROUP BY, COUNT, AVG, subqueries para top_applied e duplicates), uma página HTML vanilla, cache simples em memória. Sem novas migrações de banco. O maior risco está na consistência das queries com o schema existente (Stories 3.1–3.7).

## Riscos

- **Queries lentas em produção:** Agregações sobre `insights` sem índices específicos podem ser lentas conforme o volume cresce. Mitigação: cache com TTL 60s + queries limitadas (top_applied LIMIT 5). Índices auxiliares já existem em `status`, `source`, `confidence_level` (Story 3.1).
- **Schema de 3.6/3.7 pode adicionar campos novos:** Se Stories 3.6 e 3.7 alterarem o schema, os queries de dashboard podem precisar de ajuste. Mitigação: esta story depende de 3.6 e 3.7 Done — implementar após ambas concluídas.
- **applicationProof.improvementPercent pode ser null:** Nem todo insight aplicado terá `improvementPercent` calculado — o campo é opcional em `ApplicationProof`. A story trata isso retornando `null` sem quebrar o contrato.
- **Epic 4 pode mudar o contrato de dados:** O payload de `GET /brain/dashboard` será o contrato de dados consumido pelo Epic 4. Se Epic 4 precisar de campos adicionais, haverá uma story de extensão. Documentar isso no SOP.

## Dev Notes

### Tipo `DashboardData` (contrato de dados para Epic 4)

```typescript
// packages/brain/src/types/dashboard.ts

export interface DashboardData {
  generatedAt: string;           // ISO 8601 — timestamp da geração
  cacheHit: boolean;             // true se veio do cache em memória
  summary: DashboardSummary;
  cycle: CycleMetrics;
  insights_by_confidence: ConfidenceBreakdown;
  top_applied: TopAppliedInsight[];
  quality_distribution: QualityBucket[];
  duplicates: DuplicatesReport;
}

export interface DashboardSummary {
  total: number;
  by_status: {
    raw: number;
    validated: number;
    applied: number;
    rejected: number;
  };
  by_source: {
    zenya_operation: number;
    agent_research: number;
    mauro_input: number;
  };
  total_duplicates: number;
  avg_quality_score: number | null;  // null se não houver insights validados/aplicados
}

export interface CycleMetrics {
  ingested: number;      // total criados (todos os status)
  validated: number;     // total que chegaram a validated, applied ou além
  applied: number;       // total com status='applied'
  rejected: number;      // total com status='rejected'
  completionRate: number; // applied / ingested * 100, 2 casas decimais (0 se ingested=0)
}

export interface ConfidenceBreakdown {
  authoritative: number;
  high: number;
  medium: number;
}

export interface TopAppliedInsight {
  id: string;
  summary: string | null;
  source: string;
  confidenceLevel: string;
  appliedAt: string | null;
  improvementPercent: number | null;
  nucleusId: string | null;
}

export interface QualityBucket {
  range: string;          // ex: "0.0–0.2"
  min: number;            // ex: 0.0
  max: number;            // ex: 0.2
  count: number;
}

export interface DuplicatesReport {
  total_duplicates: number;
  top_canonical: TopCanonical[];
}

export interface TopCanonical {
  canonicalId: string;
  count: number;
  summary: string | null;
}
```

### Queries SQL recomendadas

**summary.by_status (uma query):**
```sql
SELECT
  COUNT(*) FILTER (WHERE status = 'raw')       AS raw,
  COUNT(*) FILTER (WHERE status = 'validated') AS validated,
  COUNT(*) FILTER (WHERE status = 'applied')   AS applied,
  COUNT(*) FILTER (WHERE status = 'rejected')  AS rejected,
  COUNT(*)                                      AS total,
  COUNT(*) FILTER (WHERE is_duplicate = true)  AS total_duplicates,
  ROUND(AVG(quality_score) FILTER (WHERE status IN ('validated', 'applied')), 2) AS avg_quality_score
FROM insights;
```

**summary.by_source:**
```sql
SELECT source, COUNT(*) AS count
FROM insights
GROUP BY source;
```

**cycle.validated** (soma de validated + applied — insights que completaram a fase de validação):
```sql
SELECT COUNT(*) FROM insights WHERE status IN ('validated', 'applied');
```

**top_applied:**
```sql
SELECT
  id,
  summary,
  source,
  confidence_level,
  applied_at,
  (application_proof->>'improvementPercent')::numeric AS improvement_percent,
  nucleus_id
FROM insights
WHERE status = 'applied'
ORDER BY applied_at DESC NULLS LAST
LIMIT 5;
```

**quality_distribution:**
```sql
SELECT
  CASE
    WHEN quality_score < 0.2  THEN '0.0–0.2'
    WHEN quality_score < 0.4  THEN '0.2–0.4'
    WHEN quality_score < 0.6  THEN '0.4–0.6'
    WHEN quality_score < 0.8  THEN '0.6–0.8'
    ELSE                           '0.8–1.0'
  END AS range,
  COUNT(*) AS count
FROM insights
WHERE status IN ('validated', 'applied')
  AND quality_score IS NOT NULL
GROUP BY 1
ORDER BY 1;
```

**duplicates.top_canonical:**
```sql
SELECT
  i.canonical_id,
  COUNT(*) AS count,
  c.summary
FROM insights i
JOIN insights c ON c.id = i.canonical_id
WHERE i.is_duplicate = true
  AND i.canonical_id IS NOT NULL
GROUP BY i.canonical_id, c.summary
ORDER BY count DESC
LIMIT 5;
```

### Cache em Memória (TTL 60s)

```typescript
// packages/brain/src/services/dashboard.service.ts

interface CacheEntry {
  data: DashboardData;
  expiresAt: number;
}

let cache: CacheEntry | null = null;

export async function getDashboardData(db: SupabaseClient): Promise<DashboardData> {
  const now = Date.now();
  if (cache && cache.expiresAt > now) {
    return { ...cache.data, cacheHit: true };
  }
  const data = await fetchDashboardData(db);
  cache = { data, expiresAt: now + 60_000 };
  return data;
}
```

### Architectural Notes

**Para Epic 4:**

1. **Contrato estável confirmado:** O tipo `DashboardData` e todos os seus sub-tipos estão exportados de `packages/brain/src/types/dashboard.ts`. Epic 4 pode consumir via `GET <gateway>/brain/dashboard`. A re-exportação via `@sparkle-os/brain-client` está documentada como story futura.

2. **Observação LOW — `cycle.validated` derivado, não query separada:** O campo `cycle.validated` é calculado como `statusCounts.validated + statusCounts.applied` a partir do resultado da query 1 (status counts). Isso é correto e eficiente — evita uma 7ª query. Epic 4 deve tratar este campo como "insights que completaram a fase de validação" (validated + applied), não apenas `status=validated`.

3. **Cache invalidation:** O cache é em memória com TTL 60s. Reiniciar o serviço limpa o cache. Epic 4 deve tolerar dados com até 60s de defasagem. Para dados em tempo real, a recomendação futura seria adicionar endpoint `?force=true` com invalidação manual (fora do escopo desta story).

4. **Escalabilidade de queries:** As 6 queries usam `COUNT(*) FILTER (WHERE ...)` e índices existentes em `status`, `source`, `confidence_level` (Story 3.1). Com volume crescente, a query de `quality_distribution` (sem índice em `quality_score`) pode se tornar lenta. Mitigação já prevista: cache TTL 60s. Para volume alto (>100k insights), recomendar índice em `(status, quality_score)` como melhoria futura.
