# Story 7.8 — Validação com Zenya Prime e Cutover

**Epic:** 7 — Zenya como Órgão Nativo do SparkleOS
**Status:** Draft
**Prioridade:** P1
**Executor:** @dev + @devops
**Estimativa:** M
**Depende de:** 7.3, 7.4, 7.5, 7.6, 7.7

## Descrição

**As a** operador do sistema,
**I want** validar o comportamento da Zenya no SparkleOS usando o número próprio (Zenya Prime) antes de migrar os clientes reais,
**so that** os 4 clientes em produção não sejam expostos a nenhum risco durante a transição.

---

## Acceptance Criteria

- [ ] AC1: Zenya Prime (número WhatsApp próprio do SparkleOS) configurada como tenant no banco com webhook do Chatwoot apontando para o servidor SparkleOS
- [ ] AC2: Checklist de comportamentos validados via conversas reais no número próprio: resposta básica, escalonamento, agendamento, áudio, ferramenta customizada
- [ ] AC3: 48h de operação estável da Zenya Prime sem nenhum incidente crítico antes de qualquer cutover de cliente real
- [ ] AC4: Plano de rollback documentado por cliente: comandos exatos para reverter webhook no Chatwoot de volta ao n8n se necessário
- [ ] AC5: Cutover executado cliente a cliente (não em bloco) — começando pelo de menor risco/menor volume
- [ ] AC6: n8n mantido em standby (não desativado) por mínimo 7 dias após cutover do último cliente
- [ ] AC7: 0 incidentes críticos nos 7 dias pós-cutover de cada cliente — condição para desativação do n8n

## Dev Notes

### Checklist de validação Zenya Prime (AC2)

Executar cada cenário via conversa real com o número da Zenya Prime:

```
[ ] 1. Resposta básica — mensagem simples → resposta coerente
[ ] 2. Múltiplas mensagens rápidas — debounce/queue funcionando (sem duplicata)
[ ] 3. Escalonamento — trigger "quero falar com humano" → agente-off + alerta
[ ] 4. Message chunking — resposta longa → múltiplas partes com typing
[ ] 5. Resposta em áudio — preferência audio → mensagem de voz recebida
[ ] 6. Fallback de áudio — desligar ElevenLabs → resposta em texto automática
[ ] 7. Google Calendar (se configurado) — buscar horários → resposta correta
[ ] 8. Ferramenta customizada — se Asaas ativo → cobrança criada
[ ] 9. Memória — conversa contínua entre mensagens → contexto mantido
[ ] 10. Reinício de sessão — /reset → memória limpa
```

### Configuração do webhook por cliente no Chatwoot
```
n8n atual:      POST https://fazer.ai/webhook/{id}
SparkleOS:      POST https://{vps_ip}/webhook/chatwoot
                ou POST https://zenya.sparkle.com.br/webhook/chatwoot (se domínio configurado)
```

Para cada cliente:
1. Acessar painel Chatwoot do cliente
2. Settings > Integrations > Webhooks
3. Atualizar URL do webhook para SparkleOS
4. Manter URL do n8n salva para rollback

### Plano de rollback (AC4) — documentar por cliente
```markdown
## Rollback — Cliente X

**Condição de trigger:** Incidente crítico nas primeiras 24h após cutover

**Passos:**
1. Acessar Chatwoot do cliente: [URL]
2. Settings > Integrations > Webhooks
3. Substituir URL por: https://fazer.ai/webhook/[ID_N8N]
4. Verificar que n8n responde: enviar mensagem de teste
5. Notificar cliente sobre breve instabilidade (se houve)

**Tempo estimado de rollback:** < 5 minutos
```

### Ordem de cutover (AC5)
Sugestão de ordem por menor risco:
1. Cliente em entrega pendente (ainda não tem usuários ativos) — risco mínimo
2. Cliente com menor volume de mensagens
3. Cliente mais confiante e tolerante a instabilidade
4. Cliente com maior volume — por último

### PM2 — processo persistente na VPS
```bash
# Deploy inicial na VPS
cd /var/www/sparkle/packages/zenya
pm2 start dist/worker/webhook.js --name zenya-webhook
pm2 save
pm2 startup  # garantir reinício automático
```
