## Descrição

**As a** sistema SparkleOS,
**I want** receber webhooks do Chatwoot e enfileirar mensagens por sessão com lock distribuído,
**so that** a Zenya tenha uma fundação funcional e isolada independente do n8n.

### Schema das tabelas (migration)
```sql
CREATE TABLE IF NOT EXISTS zenya_queue (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id TEXT NOT NULL,
  phone_number TEXT NOT NULL,
  message_id TEXT NOT NULL UNIQUE,
  payload JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending', -- pending | processing | done | failed
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_zenya_queue_session ON zenya_queue (tenant_id, phone_number, status);

CREATE TABLE IF NOT EXISTS zenya_session_lock (
  tenant_id TEXT NOT NULL,
  phone_number TEXT NOT NULL,
  locked_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (tenant_id, phone_number)
);
```

### Identificação do tenant no webhook
O `tenant_id` deve ser extraído do `body.account.id` (ID da conta Chatwoot) — cada conta Chatwoot corresponde a um cliente/tenant. Isso é o identificador seguro — não vem do LLM, vem da infraestrutura.

### Filtro de mensagens outgoing
Mensagens enviadas pelo próprio bot chegam com `message_type = 'outgoing'` ou `message_type = 'activity'`. Retornar `200 OK` imediatamente sem processar — caso contrário o bot entra em loop respondendo às próprias mensagens.

### Lock pattern
```typescript
// lock.ts
async function withSessionLock(tenantId: string, phone: string, fn: () => Promise<void>) {
  await db.query(
    'INSERT INTO zenya_session_lock (tenant_id, phone_number) VALUES ($1, $2) ON CONFLICT DO NOTHING',
    [tenantId, phone]
  );
  // Se já existe lock, aguarda na fila (queue processada sequencialmente)
  // finally: remove lock
}
```
