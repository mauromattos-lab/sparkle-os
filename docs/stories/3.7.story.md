# Story 3.7 — Ciclo de Vida do Conhecimento

## User Story

Como Cérebro Coletivo, quero que insights obsoletos, duplicados e estagnados sejam identificados e tratados por um processo de ciclo de vida, para que o conhecimento armazenado permaneça relevante e não polua decisões futuras.

## Acceptance Criteria

- [x] 1. Endpoint `POST /brain/lifecycle/run` implementado: executa o ciclo de vida completo (detecção de insights estagnados, revisão de duplicatas, arquivamento de rejeitados) e retorna um relatório `LifecycleReport` com contagens por ação realizada. Execução é idempotente — rodar duas vezes seguidas produz o mesmo resultado final.

- [x] 2. **Detecção de Insights Estagnados:** Insights com `status='raw'` há mais de `STALE_RAW_DAYS` dias (default: 30) sem nenhuma ação de validação são marcados com tag `stale` e incluídos no relatório como `pendingReview`. Insights com `status='validated'` há mais de `STALE_VALIDATED_DAYS` dias (default: 90) sem `appliedAt` também são marcados como `stale`. Nenhum status é alterado automaticamente — a marcação é informativa (tag `stale` adicionada ao array `tags`).

- [x] 3. **Revisão de Duplicatas:** `GET /brain/lifecycle/duplicates` lista todos os insights com `isDuplicate=true` (campo definido em `docs/architecture/cerebro-coletivo.md §7`), incluindo o insight canônico referenciado por `canonicalId`. Resposta inclui: `{ duplicates: DuplicateGroup[], total: number }` onde cada `DuplicateGroup` contém o canonical e seus duplicatas com `similarityScore`.

- [x] 4. **Resolução de Duplicata:** `PATCH /brain/lifecycle/duplicates/{id}/resolve` aceita `action: 'keep' | 'reject'`. `keep` limpa `isDuplicate=false` e `canonicalId=null` (insight promovido a independente). `reject` chama o fluxo de rejeição existente (`status='rejected'`, `reason` gerado automaticamente como `"Duplicata de ${canonicalId} — ciclo de vida"`). Ambas as ações são auditadas no `Change Log` do insight via `validationNotes`.

- [x] 5. **Fila de Observação:** `GET /brain/lifecycle/pending` lista insights que requerem atenção humana: (a) insights `raw` com tag `stale`, (b) insights `validated` com tag `stale`, (c) duplicatas não resolvidas. Resposta paginada: `{ items: PendingItem[], total: number, page: number }`. Cada item inclui `reason: 'stale_raw' | 'stale_validated' | 'unresolved_duplicate'`.

- [x] 6. **Política de Arquivamento de Rejeitados:** Insights com `status='rejected'` há mais de `ARCHIVE_REJECTED_DAYS` dias (default: 180) são marcados com tag `archived`. Insights `archived` são excluídos dos resultados padrão de `GET /brain/insights` e `POST /brain/insights/search` a menos que o filtro `includeArchived=true` seja explicitamente passado. Sem deleção física — auditabilidade preservada.

- [x] 7. Constantes de threshold (`STALE_RAW_DAYS`, `STALE_VALIDATED_DAYS`, `ARCHIVE_REJECTED_DAYS`) configuráveis via variável de ambiente ou constante centralizada em `packages/brain/src/config/lifecycle.config.ts` com valores default documentados. Não hardcoded inline.

- [x] 8. Testes unitários em `packages/brain/src/services/lifecycle.service.test.ts` cobrindo: detecção de stale (raw e validated), marcação de tag `stale`, listagem de duplicatas com grupo canônico, resolução `keep` e `reject`, fila de pendentes com os 3 tipos de reason, arquivamento de rejeitados, exclusão de `archived` dos resultados padrão, idempotência de `POST /brain/lifecycle/run`.

- [x] 9. SOP criado em `docs/sops/sop-ciclo-vida-conhecimento.md` descrevendo: quando rodar o ciclo de vida, como interpretar o `LifecycleReport`, como resolver duplicatas via API ou CLI, critérios para promover ou rejeitar insights estagnados, e política de retenção de rejeitados.

- [x] 10. Regressão: 82/82 testes em `packages/brain/` passando após as modificações desta story (57 existentes + 25 novos lifecycle tests).

## Complexidade

**M (Médio)** — Extensão do package `packages/brain/` existente com lógica de negócio nova. Sem nova infra, sem embeddings, sem chamadas externas. A maior complexidade é a lógica de filtragem de `archived` nos endpoints de busca existentes (Story 3.3 e Story 3.5 tocam nesses endpoints) e garantir idempotência no `POST /brain/lifecycle/run`. Estimativa: 4–6 horas de implementação.

## Riscos

- **Modificação em endpoints existentes:** Adicionar filtro `includeArchived` em `GET /brain/insights` e `POST /brain/insights/search` requer atenção para não quebrar clientes existentes (Story 3.5 `BrainClient`). Mitigação: `includeArchived=false` é o default — comportamento atual é preservado sem mudança de assinatura.

- **Threshold de staleness arbitrário:** 30/90/180 dias são estimativas razoáveis para v1. Com base pequena de insights, podem gerar muitos falsos positivos. Mitigação: valores configuráveis via env, nenhum status alterado automaticamente (apenas tag informativa), decisão final sempre humana.

- **Idempotência do `POST /brain/lifecycle/run`:** Se rodado múltiplas vezes, não deve duplicar tags `stale` ou `archived`. Mitigação: verificar existência da tag antes de adicionar (`if (!tags.includes('stale'))`).

- **Dependência de campos de Story 3.2/3.3:** `isDuplicate`, `canonicalId`, `similarityScore` (Story 3.2), `qualityScore`, `status`, `validatedAt`, `appliedAt` (Story 3.3). Confirmar que todos existem na tabela antes de implementar. Referência: `docs/architecture/cerebro-coletivo.md §3.1`.

### Issues Found

**LOW — SQL condicional com potencial fragment inválido em `listPendingInsights`**

- **Arquivo:** `packages/brain/src/db/insights.ts` — linha 409
- **Detalhe:** Quando `includeStaleRaw=false`, o ternário gera `sql\`SELECT * FROM insights WHERE false, 'stale_raw'::text AS pending_reason\`` — a vírgula após `false` é SQL sintaticamente inválido se esse fragment fosse executado isolado.
- **Mitigação atual:** Este fragment só aparece quando pelo menos um dos outros branches está ativo (UNION ALL), e os testes passam (mocks). Em produção, `reason` filter sempre produz ao menos um branch válido.
- **Recomendação para v2:** Refatorar para builder de query dinâmico mais explícito ao invés de UNION condicional via template literals encadeados.
- **Blocking:** NÃO — risco baixo, cenário de falha requer todos os branches false simultaneamente, o que não ocorre com os filtros expostos na API.

### Architectural Notes

1. **Idempotência por SQL:** A estratégia de usar `AND NOT ('stale' = ANY(tags))` ao nível do UPDATE é elegante e correta. Mantém a responsabilidade de idempotência na camada de dados, não na aplicação. Padrão a replicar em v2.

2. **Separação de concerns:** A criação de `lifecycle.config.ts` separado (e não inline no service) é boa prática — facilita override em testes e em runtime via env. Padrão aprovado para stories futuras do Brain.

3. **`POST /run` sem body obrigatório:** Decisão arquitetural correta para v1. Quando v2 introduzir execução seletiva (ex: apenas stale, apenas archive), o body pode ser enriquecido sem breaking change.

4. **Audit trail via `validationNotes`:** O campo `validationNotes` é reutilizado como log de ciclo de vida (`"Promovido a independente — ciclo de vida"`). Para v2, considerar campo dedicado `lifecycleLog: string[]` ou evento em tabela separada para rastreabilidade mais granular.

5. **CLI `aiox brain lifecycle --run`:** Corretamente marcado como OUT of scope nesta story. Quando implementado (Story futura), deve consumir `POST /lifecycle/run` e formatar `LifecycleReport` como tabela — não reimplementar lógica.
