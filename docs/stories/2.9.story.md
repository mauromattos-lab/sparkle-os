# Story 2.9 — Protocolo de Erro e Fallback da Zenya

## User Story
Como Mauro, quero que a Zenya lide com falhas de forma controlada — quando um serviço externo cai, quando um fluxo falha, quando a IA não responde — para que clientes não fiquem sem resposta e eu seja notificado apenas quando a intervenção humana for realmente necessária.

## Complexidade
**M (Médio)** — Mapeamento é analítico (deriva da Story 2.1); health check e classes de erro são implementação focada; o ponto mais elaborado é o SOP de incidentes que requer raciocínio sobre severidades.

## Riscos
- n8n não expõe API de saúde granular — o health check pode ser apenas um `GET /api/v1/healthz`; documentar limitação
- O comportamento de fallback nos fluxos n8n (ex: o que acontece se OpenAI cai) depende de como os fluxos foram construídos — pode não ser possível controlar sem modificar os fluxos

### Health Check Endpoint
```typescript
// GET /health → verifica serviços e retorna:
{
  status: 'healthy' | 'degraded' | 'down',
  services: {
    n8n: { status: 'up' | 'down', latencyMs: number },
    chatwoot: { status: 'up' | 'down', latencyMs: number },
    postgres: { status: 'up' | 'down', latencyMs: number }
  },
  checkedAt: string
}
```
- `healthy`: todos os serviços up
- `degraded`: pelo menos 1 serviço down (mas n8n + Chatwoot ainda up)
- `down`: n8n ou Chatwoot down (operação da Zenya impossível)

### Critérios de Notificação (para documentar no SOP)
- **Sempre notificar Mauro:** Postgres down, n8n down por >5 min, erro P1 sem recuperação automática
- **Tentar antes de escalar:** Timeout de serviço externo (retry 3x com backoff), erro de parsing de resposta
