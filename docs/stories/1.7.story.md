# Story 1.7 — Framework de SOPs

## User Story
Como agente AIOS, quero um template e processo para documentar SOPs, para que todo processo repetível criado no SparkleOS tenha instruções claras de como replicá-lo.

## Status
- [x] Draft
- [ ] Ready (pendente validação @po)
- [ ] InProgress
- [ ] InReview
- [ ] Done

## Acceptance Criteria
- [ ] 1. Template de SOP definido com estrutura mínima: objetivo, pré-requisitos, passos, resultado esperado, responsável
- [ ] 2. Localização dos SOPs no repositório definida: `docs/sops/sop-{slug}.md`
- [ ] 3. Primeiro SOP criado: `docs/sops/sop-criar-sop.md` — documenta o próprio processo de criação de SOPs
- [ ] 4. Regra formal documentada e integrada ao story lifecycle: toda story que cria processo repetível inclui SOP como AC obrigatório no Definition of Done
- [ ] 5. SOPs consultáveis por agentes em qualquer sessão — listagem acessível

## Technical Context

### Template de SOP (conteúdo de `SOP-TEMPLATE.md`)

```markdown
# SOP-{SLUG} — Título do Processo

**Versão:** 1.0
**Data:** YYYY-MM-DD
**Autor:** @agent-id
**Story de origem:** X.Y
**Revisão:** @qa (PASS/NEEDS WORK)

## Objetivo
[O que este processo faz e por que existe.]

## Pré-requisitos
- [ ] Requisito 1
- [ ] Requisito 2

## Responsável
[Quem executa este processo — agente ou Mauro]

## Passos

### Passo 1 — [Título]
[Descrição detalhada. Incluir comandos exatos quando aplicável.]

```bash
# exemplo de comando
```

**Resultado esperado:** [O que deve acontecer após este passo]

### Passo 2 — [Título]
...

## Resultado Final
[Estado do sistema após o processo completo. Como verificar que funcionou.]

## Troubleshooting
| Problema | Causa Provável | Solução |
|----------|---------------|---------|
| ...      | ...           | ...     |

## Histórico de Revisões
| Data | Versão | Mudança | Autor |
|------|--------|---------|-------|
| ...  | 1.0    | Criação | @sm   |
```

### Integração com story lifecycle
A regra de SOP obrigatório deve ser integrada ao template de story e ao QA gate:
- No Definition of Done de qualquer story que cria processo repetível: `- [ ] SOP criado em docs/sops/`
- @qa verifica presença do SOP antes de dar PASS em stories de processo
