# Story 1.7 — Framework de SOPs

## User Story
Como agente AIOS, quero um template e processo para documentar SOPs, para que todo processo repetível criado no SparkleOS tenha instruções claras de como replicá-lo.

## Status
- [x] Draft
- [x] Ready (GO — @po Pax, 2026-04-11)
- [x] InProgress
- [x] InReview
- [x] Done

## Acceptance Criteria
- [x] 1. Template de SOP definido com estrutura mínima: objetivo, pré-requisitos, passos, resultado esperado, responsável
- [x] 2. Localização dos SOPs no repositório definida: `docs/sops/sop-{slug}.md`
- [x] 3. Primeiro SOP criado: `docs/sops/sop-criar-sop.md` — documenta o próprio processo de criação de SOPs
- [x] 4. Regra formal documentada e integrada ao story lifecycle: toda story que cria processo repetível inclui SOP como AC obrigatório no Definition of Done
- [x] 5. SOPs consultáveis por agentes em qualquer sessão — listagem acessível via `docs/sops/README.md`

## Escopo

**IN (esta story entrega):**
- `docs/sops/SOP-TEMPLATE.md` — template padrão de SOP
- `docs/sops/sop-criar-sop.md` — meta-SOP (como criar SOPs)
- `docs/sops/README.md` — índice de todos os SOPs existentes com links
- Regra formal documentada no lifecycle: stories com processos repetíveis exigem SOP no DoD

**OUT (não inclui):**
- Indexação de SOPs no Postgres (→ futuro, estrutura preparada para isso)
- Criação de novos SOPs além dos especificados (cada story cria seus próprios)
- Modificação dos SOPs já criados nas stories 1.2–1.6 (exceto verificar conformidade com template)

## Technical Context

### Template de SOP (conteúdo de `SOP-TEMPLATE.md`)

```markdown
# SOP-{SLUG} — Título do Processo

**Versão:** 1.0
**Data:** YYYY-MM-DD
**Autor:** @agent-id
**Story de origem:** X.Y
**Revisão:** @qa (PASS/NEEDS WORK)

## Objetivo
[O que este processo faz e por que existe.]

## Pré-requisitos
- [ ] Requisito 1
- [ ] Requisito 2

## Responsável
[Quem executa este processo — agente ou Mauro]

## Passos

### Passo 1 — [Título]
[Descrição detalhada. Incluir comandos exatos quando aplicável.]

```bash
# exemplo de comando
```

**Resultado esperado:** [O que deve acontecer após este passo]

### Passo 2 — [Título]
...

## Resultado Final
[Estado do sistema após o processo completo. Como verificar que funcionou.]

## Troubleshooting
| Problema | Causa Provável | Solução |
|----------|---------------|---------|
| ...      | ...           | ...     |

## Histórico de Revisões
| Data | Versão | Mudança | Autor |
|------|--------|---------|-------|
| ...  | 1.0    | Criação | @sm   |
```
