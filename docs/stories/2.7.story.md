# Story 2.7 — Isolamento de Dados por Cliente na Zenya

## User Story
Como Mauro, quero garantir que os dados de conversa de cada cliente atendido pela Zenya sejam isolados por Row-Level Security no Postgres — para que seja impossível, por bug ou erro de query, que dados de um cliente vazem para outro.

## Complexidade
**M (Médio)** — Implementação de RLS no Postgres é bem definida, mas testar o isolamento requer atenção. A integração com o Zenya Adapter (configurar `app.current_client_key` nas queries) é o ponto mais crítico.

## Riscos
- Supabase pode ter restrições na configuração de `set_config` via pool de conexões — verificar se `current_setting` funciona corretamente com o pooler do Supabase (Session vs. Transaction mode)
- Drizzle ORM pode precisar de adaptação para injetar `set_config` antes de cada query — verificar suporte nativo ou criar middleware

## Dev Notes

### Schema e RLS
[Source: docs/architecture.md §9]
```sql
CREATE TABLE zenya_conversations (
  id                 UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  client_id          UUID NOT NULL REFERENCES zenya_clients(id),
  isolation_key      TEXT NOT NULL,
  chatwoot_conv_id   INTEGER,
  content            JSONB NOT NULL DEFAULT '{}',
  created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
ALTER TABLE zenya_conversations ENABLE ROW LEVEL SECURITY;
CREATE POLICY zenya_client_isolation ON zenya_conversations
  USING (isolation_key = current_setting('app.current_client_key', TRUE));
```

### Padrão de Uso em Runtime
Para queries em `zenya_conversations`, o código deve:
```typescript
// Antes de qualquer query nesta tabela:
await db.execute(
  sql`SELECT set_config('app.current_client_key', ${isolationKey}, TRUE)`
);
// Agora as queries respeitam o RLS automaticamente
const conversations = await db.select().from(zenyaConversations);
```

### Referência: `dataIsolationKey`
[Source: docs/architecture.md §4 — ZenyaClient]
```typescript
dataIsolationKey: string; // chave usada no RLS — gerada no provisionamento
```
A chave é gerada pelo Core no provisionamento (`POST /nucleus/zenya/clients`, Story 2.4) e armazenada em `zenya_clients.data_isolation_key`. Esta mesma chave é inserida em `zenya_conversations.isolation_key` ao salvar conversas.

### Supabase Connection Mode
[Source: docs/architecture.md §2.2]
- Supabase free tier usa Transaction mode por padrão no pooler
- `set_config` com `is_local=TRUE` funciona por transação — compatível com Transaction mode
- Usar `set_config('app.current_client_key', key, TRUE)` (terceiro param = is_local)

### Padrão Drizzle do packages/core
[Source: packages/core/src/db/ — padrão Epic 1]
- Drizzle com `postgres` driver já em uso no Core
- `organs/zenya/src/db/` deve seguir o mesmo padrão de schema + migrations
