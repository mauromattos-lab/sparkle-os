## Descrição

Liberar o prompt refinado da Julia (v4, com horário de atendimento humano) em produção e acompanhar o comportamento via feedback direto dela. Esta é a fase 4 do ciclo brownfield do [`TENANT-REFINEMENT-PLAYBOOK.md §2 passo 5-7`](../zenya/TENANT-REFINEMENT-PLAYBOOK.md) (whitelist → release → monitorar).

**Diferença entre o playbook e a execução real:**
O playbook idealiza "janela agendada + whitelist formal + monitoramento 96h explícito". Com a Julia, o processo foi mais orgânico — ela já estava atendendo clientes com a Zenya em produção há semanas. As versões v2, v3 e v4 foram aplicadas diretamente em produção (cliente ativo consentiu via comunicação direta com Mauro) e o "monitoramento" foi contínuo via WhatsApp dela reportando o que funcionava ou não.

Isso é **aceitável** para este caso específico:
- Cliente era a própria Julia (dona do negócio) — consentimento direto
- Volume de tráfego baixo o suficiente pra Mauro acompanhar caso a caso
- Fixes aplicados eram de melhoria, não de correção crítica (bot já funcionava)

Para tenants futuros em brownfield **o playbook formal deve ser seguido** (whitelist explícita + janela negociada + 96h) — este caso é exceção documentada, não precedente.

## Lições para o playbook (input pra Epic 15 v2)

- A fase 4 do playbook assume "janela + whitelist + 96h" como regra. Com cliente pequeno e engajado (dono do negócio como único atendente), esse overhead não é necessário — **adicionar no playbook uma seção "Variante para tenant de baixo volume com cliente engajado"** que autoriza execução orgânica desde que (a) cliente consente diretamente, (b) Mauro acompanha caso a caso, (c) rollback SQL pronto.
- Iteração direta em produção com feedback da cliente foi **mais eficaz** que REPL+smoke pra fixes de prompt onde o valor é subjetivo ("não quero resumo, não gostei"). Adicionar ao playbook que **preferências estéticas do cliente só se validam em produção real**, não em smoke.
